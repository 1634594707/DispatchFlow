# JD 可能追问的问题与参考回答

## 1. 你为什么做这个项目

### 问题

为什么会做一个无人车调度系统原型，而不是普通管理后台项目？

### 参考回答

我想做的不是单纯的信息管理系统，而是带有实时调度、状态流转和设备协同的后端系统。无人车调度这个方向同时包含订单、任务、车辆、状态机和事件驱动，业务复杂度和工程复杂度都更高。我做 DispatchFlow 的目的，就是先把这类系统的核心后端链路做出来，而不是停留在页面展示层。

## 2. 这个项目的核心业务闭环是什么

### 问题

你这个系统最核心的业务流程是什么？

### 参考回答

核心闭环是：

`下单 -> 创建订单 -> 创建调度任务 -> 自动派车 -> 车辆执行 -> 车辆回报 -> 任务完成或失败 -> 回待命位或去充电`

我重点做的是把这条链路在后端真正接起来。移动端下单不会直接改车辆，而是先创建订单，再生成调度任务，再触发自动派车。车辆也不是瞬间送达，而是通过状态机逐步推进，最后由车辆回报驱动任务状态变化。

## 3. 自动派车是怎么做的

### 问题

你的自动派车逻辑是什么？

### 参考回答

自动派车的核心在 `DispatchTaskServiceImpl.autoAssignTask`。它的流程是：

1. 先获取任务锁，避免并发重复派车
2. 校验任务状态是否允许自动派车
3. 查询可分配车辆
4. 根据取货点调用选车逻辑
5. 占用车辆
6. 任务状态进入 `ASSIGNED`
7. 如果没有可分配车辆或发生冲突，就进入 `MANUAL_PENDING`

当前项目实现的是最近车策略，但不是只算直线距离，而是优先按路径规划后的路网距离选车。

## 4. 为什么要加任务锁

### 问题

为什么自动派车时要加锁？

### 参考回答

因为调度任务可能被重复触发分配，比如接口重试、定时补偿或者多个线程同时处理。如果不加锁，同一个任务可能会被多次派给不同车辆。这个项目里我用 `DispatchLockService` 做了任务级锁抽象，目的是保证同一时刻只有一个派车流程能处理某个任务。

## 5. 车辆状态机为什么重要

### 问题

为什么不用“任务创建后直接完成”，而要做状态机？

### 参考回答

因为真实无人车调度不是一个瞬时动作，而是一个持续过程。车辆会经历前往取货、装货、前往送货、卸货、返航或充电这些阶段。如果没有状态机，系统很难表达任务真实进度，也无法合理处理低电量、离线、人工介入等场景。

我在项目里把这些逻辑放在 `ParkPilotSimulationServiceImpl`，让车辆按 tick 周期推进，而不是直接跳到终态。

## 6. 低电量是怎么处理的

### 问题

送货完成后，如果车快没电了怎么办？

### 参考回答

我在任务完成后的后处理里做了电量判断。核心逻辑是：

- 如果车辆电量低于阈值，就从送货完成直接转去充电
- 如果电量正常，就返回待命位

这个逻辑放在 `routeAfterDelivery`，目的是让调度系统不只关注“送完货”，还关注车辆生命周期和后续可用性。

## 7. 为什么停车位和充电位统一

### 问题

你为什么把停车位和充电位做成同一套点位？

### 参考回答

这是为了更贴近当前项目要模拟的园区场景。很多短驳无人车场景里，车辆待命时就在停车补能区，不需要单独切一套完全独立的充电区。统一建模后，大屏表达、移动端预览和后端调度逻辑会更一致，也减少了无意义的状态切换复杂度。

## 8. 车辆执行状态是怎么变化的

### 问题

任务状态和车辆状态怎么联动？

### 参考回答

联动分两层：

第一层是运行时状态机，控制车辆当前处于 `TO_PICKUP`、`LOADING`、`TO_DROPOFF` 这些阶段。

第二层是车辆回报驱动任务状态变化。比如：

- `START_EXECUTE` 把任务推进到 `EXECUTING`
- `TASK_SUCCESS` 把任务推进到 `SUCCESS`
- `TASK_FAILED` 把任务推进到 `FAILED`

我把这层统一放在 `VehicleReportServiceImpl`，这样任务状态变化是由执行回报驱动，而不是后端随意改状态。

## 9. RabbitMQ 在这个项目里干什么

### 问题

你项目里为什么要用 RabbitMQ？

### 参考回答

RabbitMQ 主要用来承接调度域里的关键业务事件，比如：

- 任务创建
- 任务已分配
- 任务执行中
- 任务成功
- 任务失败
- 异常创建和异常解决

这样做的意义是把业务动作和后续消费解耦，让系统更适合扩展审计、告警、异步通知和后续其他服务接入。

## 10. 为什么不用直接发消息，而要用 Outbox

### 问题

为什么不在业务代码里直接发 RabbitMQ，而要加 Outbox？

### 参考回答

因为业务事务提交和 MQ 发送不是天然一致的。如果订单或任务已经更新成功，但消息发送失败，就会产生状态不一致问题。这个项目里我用了 Outbox 模式：

1. 先把事件落本地消息表
2. 事务提交后再发 MQ
3. 发送失败标记为失败
4. 定时任务重试

这样可以降低业务状态和消息投递不一致的风险。

## 11. 你项目里 Redis 可以怎么用

### 问题

Redis 在你项目里主要适合做什么？

### 参考回答

当前最适合 Redis 的场景有三个：

1. 任务分配锁
2. 车辆回报幂等
3. 事件消费幂等

也就是说 Redis 在这里不是简单做缓存，而是用来解决调度系统里的并发保护和幂等控制问题。

## 12. 如果没有可用车辆怎么办

### 问题

自动派车失败时怎么处理？

### 参考回答

我没有让系统直接报错结束，而是让任务进入 `MANUAL_PENDING`，并记录失败原因。同时会创建异常记录，方便后续人工介入。这样设计是因为真实调度系统里“自动化失败后有人接手”是很常见的情况。

## 13. 如果后续做成微服务，你会怎么拆

### 问题

如果你们要上 Spring Cloud，这个项目怎么拆？

### 参考回答

我会优先按业务边界拆：

- 订单服务
- 调度服务
- 车辆服务
- 管理端聚合 API 或网关层

其中调度服务保留任务、异常、事件、选车策略和状态机。订单服务只负责订单生命周期，车辆服务只负责车辆状态和车辆回报。然后再按需要接入注册发现、配置中心和网关。

## 14. 这个项目最难的点是什么

### 问题

你觉得这个项目最难的部分是什么？

### 参考回答

我觉得最难的是把几个概念真正接成闭环，而不是分别做出来。

比如：

- 下单不能只停留在前端，要真正落到订单和任务
- 自动派车不能只选车，还要考虑并发和失败降级
- 状态机不能只会动，还要和任务状态、车辆回报、电量策略结合
- RabbitMQ 不能只配出来，还要考虑事务和消息一致性

也就是说，难点不是单个模块，而是后端闭环是否成立。

## 15. 你最想参与哪个模块

### 问题

如果进团队，你最想参与哪个模块？

### 参考回答

我最想参与调度核心服务，尤其是选车策略、状态机和异常重调度这部分。因为这部分最能体现系统业务价值，也最能把后端设计、并发控制和事件驱动结合起来。
