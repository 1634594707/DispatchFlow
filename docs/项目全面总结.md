## 一、项目定位与整体目标

### 1.1 业务背景

DispatchFlow 是面向**园区短驳配送场景**的无人车调度平台。真实落地场景为**南通海门叠石桥国际家纺城**（ZJF = 找家纺），覆盖 L4 无人车在真实园区内的人车混行、窄路、笼车 90 秒装卸等场景。整个系统从订单产生到车辆执行、再到完成回充形成端到端闭环。

### 1.2 系统目标

| 维度 | 目标 |
| --- | --- |
| 业务闭环 | 移动端下单 → 自动派车 → 车辆执行 → 实时监控 → 异常处置 → 运营分析 |
| 调度智能化 | 多维度评分选车、SOC 全链路估算、MAPF 多车冲突避免、路线碰撞校验 |
| 地图精准化 | L0/L1/L2 三层地图、GCJ-02 坐标系、车辆 8 状态可视化、建筑 Polygon 障碍物膨胀 |
| 集成开放性 | SIM 仿真 / REAL HTTP 网关 / VDA5050 MQTT 三种车辆链路统一适配 |
| 工程化标准 | 前后端分离、Docker Compose 部署、Flyway 迁移、CI/CD 流水线、质量门禁 |
| 多端覆盖 | 管理端（PC）+ 监控大屏 + 移动端下单 + PWA 安装 + Android 原生壳 |

### 1.3 在线演示

- 访问地址：[aplicity.online](https://www.aplicity.online)
- 移动端：[app.aplicity.online](https://app.aplicity.online)
- 三个默认账号：admin / operator / viewer（详见 README）

### 2.2 订单与任务模块

**职责**：管理从下单到完成的订单生命周期与调度任务流转。

| 功能点 | 作用 |
| --- | --- |
| 移动端下单 | 商家通过移动端选择取货点/送货点/线路，创建订单（`POST /api/admin/park/orders`，需 `X-Mobile-Api-Key`） |
| 订单查询 | 列表/详情/取消（管理端 + 移动端"我的订单"） |
| 订单状态机 | `WAITING_DISPATCH → DISPATCHED → DELIVERED`（含 `CANCELLED`） |
| 任务创建 | 订单创建后自动生成 `DispatchTaskEntity`（status=PENDING） |
| 任务派单 | 自动派车（评分选车）+ 手动派车（指定车辆）+ 改派 + 批量撤销 + 优先级提升 |
| 任务状态机 | 8 状态：`PENDING → ASSIGNING → ASSIGNED → EXECUTING → SUCCESS/FAILED/CANCELLED`，含 `MANUAL_PENDING`（自动失败转人工） |
| 任务操作日志 | `t_dispatch_task_operate_log` 记录所有状态变更（含操作人/前后状态） |
| 任务池 | 服务端分页 + 手动排序（前端 localStorage 持久化） |

| 功能点 | 作用 |

### 2.5 园区基础设施模块

**职责**：园区物理世界的数字化建模，是调度可行性的基础。

| 功能点 | 作用 |
| --- | --- |
| 园区管理 | 多园区支持，含坐标系锚点、地图尺寸、米/像素比 |
| 站点管理 | A/B/C 三级可信度，含 `anchorNodeCode`（接入路网）、`serviceDirection`、`stationConfidence` |
| 服务位管理 | 站点子表 `t_station_service_position`，精确控制车辆到达位置、朝向（`stopHeading`）、进出方向 |
| 服务位锁 | 瞬时锁 + 带过期预约 + 扫描过期 + 释放原因（COMPLETED/TIMEOUT/CANCELLED/REASSIGN） |
| 路网管理 | 节点 + 路段，含 `roadClass`（HIGHWAY/ARTERIAL/.../FIRE_LANE）、`accessState`、`direction`、时间窗封路 |
| 停车位管理 | `facingDirection`、`entryNodeCode/exitNodeCode`、`blockingMainRoad` 标记 |
| 充电桩管理 | `plugType`（CCS2/GB_T_DC/CHAOJI/AC_GENERIC/WIRELESS）、`entryNodeCode/exitNodeCode`、预约状态 |
| 换电柜管理 | 换电柜 CRUD |
| 地理围栏 | `fenceType`（BOUNDARY 出界告警 / RESTRICTED 进入告警）+ `responseLevel`（INFO/WARN/BLOCK）+ `bufferMeters` |
| 建筑障碍物 | 真实 GeoJSON Polygon + `blockType`（BUILDING/WALL/RIVER/...）+ `isHardForbidden` + `defaultExpansionBufferMeters` |
| 地图版本管理 | `MapDataVersion` 支持多版本切换 + 激活 + checksum |
| 交通管控 | 拥堵刷新、路段禁用/降级、pause-zones 暂停区域 |
| 园区配置向导 | 5 步引导：园区 → 站点 → 路网 → 充电 → 试派单 |

### 2.6 路线安全模块

**职责**：保证车辆实际行驶路线安全可行，避免穿越建筑/水域/禁行区。

| 功能点 | 作用 |
| --- | --- |
| 起终点吸附 | `RouteEndpointSnapper` 将订单坐标吸附到最近道路节点（默认 50m 阈值），超阈值返回 `START_OFF_ROAD/END_OFF_ROAD` |
| 路线规划 | `ChainedRoadRouteService` 链式调用：本地路网（Dijkstra）→ 高德驾车 → 直线兜底（可禁用） |
| 两层碰撞校验 | 中心线碰撞 + 车辆包络膨胀碰撞（按 `safetyBufferMeters` 膨胀建筑 Polygon） |
| 直线回退拦截 | `STRAIGHT_LINE` 路线源在 `allowStraightLine=false` 时强制 `invalid` |
| 路线审计 | `t_route_audit` 记录规划/实际 polyline、长度、偏航、重规划次数 |
| 路线健康指标 | `t_route_health_metric` 统计 `EMPTY_ROAD_NETWORK/DISCONNECTED/NOT_SNAPPED/OFF_ROAD/CROSSES_BUILDING` 等指标 |
| 不可达原因细分 | 15 种 `RouteUnreachableReason` 枚举 |

### 2.12 Fleet 集成模块

**职责**：统一适配三种车辆链路，屏蔽接入差异。

| 链路模式 | 数据来源 | 关键类 |
| --- | --- | --- |

### 2.13 模块协作关系

模块间通过**事件驱动 + 同步调用**两种方式协作：

```text
[移动端下单] ──HTTP──> [订单模块] ──同步──> [任务模块]
                                              │
                                              ▼ (Outbox 事件)
                                    [RabbitMQ 事件总线]
                                              │
                ┌─────────────────────────────┼─────────────────────────────┐
                ▼                             ▼                             ▼
        [SSE 推送前端]              [Fleet 适配器下发指令]         [Webhook 外部系统]
                                              │
                                              ▼
                                      [车辆执行任务]
                                              │
                                              ▼ (遥测回报)
                              [Fleet 运行态 Redis] ──> [围栏评估]
                                              │
                                              ▼
                                [任务状态推进] ──> [异常处置]
                                              │
                                              ▼
                                      [运营分析归档]
```

**关键协作点**：

1. **订单 → 任务**：订单创建同步生成任务，任务派单同步更新订单状态
2. **任务 → 车辆**：通过 `VehicleCommandService` 下发命令，REAL 走 HTTP 轮询，VDA5050 走 MQTT 发布
3. **车辆 → Fleet**：遥测回报写入 Redis 运行态 + 触发围栏评估 + 持久化轨迹点
4. **Fleet → 前端**：通过 SSE 推送 `VEHICLE_TELEMETRY/TASK_STATUS_CHANGED/EXCEPTION_RAISED` 事件
5. **路线安全 → 调度**：派单前调用 `isReachable` 校验可达性，路线规划后调用碰撞校验
6. **园区设施 → 路网**：`ParkRoadGraph.fromDatabase()` 动态加载节点/路段/建筑，按时间窗过滤
7. **能源 → 调度**：派单评分包含 SOC 全链路校验（含回充距离），低电车辆自动返充

### 3.4 关键算法与业务流程

#### 3.4.1 派单算法（核心）

**入口**：`DispatchVehicleAssignServiceImpl.selectBestVehicle(order)`

**8 阶段流程**：

```text
订单进入
  │
  ▼
1. 前置校验
   ├─ 园区是否暂停派单？ (DispatchPauseControlService)
   ├─ 取货/送货站点是否存在且归属正确？ (ParkStationService)
   ├─ 自动绑定 deliveryZone (GEO_DELIVERY / SCHEMATIC)
   └─ 解析能量/评分策略 (DispatchStrategyRuntimeService)
  │
  ▼
2. 容量校验（7 个 checkpoint）
   ├─ 枢纽/母站容量 (HubCapacityService)
   ├─ 路线服务时间窗 (DispatchRouteService.isRouteWithinServiceWindow)
   ├─ 路线并发上限 (isRouteOccupancyAvailable)
   └─ 交通管控区 (TrafficZoneControlService.isPointInPausedZone)
  │
  ▼
3. 候选车辆过滤（5 层链式 filter）
   ├─ SOC ≥ minAssignableSoc（默认 30）
   ├─ 非维保中
   ├─ 匹配订单所需车型
   ├─ 匹配 Fleet（PilotFleetSupport.matchesOrderFleet）
   ├─ 匹配配送区域（deliveryZone）
   └─ 匹配载重（maxLoadCapacity ≥ order.weight）
  │
  ▼
4. SOC 全链路校验（canCompleteTaskWithSoc）
   公式：consumedSoc = (pickupDist + dropoffDist + returnToChargerMeters) × busyDrainMetersPerPercent
   通过条件：soc - consumedSoc ≥ minAssignableSoc
   亮点：包含"送货完成→回充电桩"距离，防止送完货无法回充
  │
  ▼
5. 可达性扫描
   对每辆候选车调 ParkRoutePlannerService.isReachable + buildRoute
   不可达车辆剔除
  │
  ▼
6. 距离混合（DispatchGeoDistanceService.applyGeoBlend）
   本地路网距离 + 高德物流距离按 blendWeight 混合
  │
  ▼
7. 评分排序（scoreCandidate）
   total = (distanceScore + socScore - pluggedBonus - idleBonus) × priorityFactor
   ├─ distanceScore = distance × weightDistance（高峰 ×0.85）
   ├─ socScore = (fullSoc - soc) × weightSocMargin（高峰 ×0.7）
   ├─ pluggedBonus：插电+STANDBY+满电时奖励
   ├─ idleBonus：空闲时长奖励（公平性）
   └─ priorityFactor：HIGH=0.7 / NORMAL=1.0 / LOW=1.3（值越小越优先）
  │
  ▼
8. MAPF 冲突预约（selectWithMapfReservation）
   按评分升序，依次尝试 mapfRoutePlannerService.planAndReserve
   第一个预约成功的车辆胜出；全部失败降级为评分第一名
```

**失败码**：`NO_VEHICLE`、`LOW_SOC`、`UNREACHABLE`、`HUB_CAPACITY_FULL`、`ROUTE_OCCUPANCY_FULL`、`CONFLICT`。

#### 3.4.2 路网建模与寻路（ParkRoadGraph）

**数据结构**：

```java
class ParkRoadGraph {
    Map<String, NodeView> nodes;              // 节点编码 → 节点视图
    Map<String, List<String>> adjacency;      // 邻接表
    Map<String, Double> edgeCostMultiplier;   // 边代价乘子（key = "from>to"）
}
```

**构建流程**（`fromDatabase(nodes, segments, now)`）：

1. 加载所有 `ACTIVE` 节点
2. 加载所有 `ACTIVE` 路段
3. **通行语义过滤**：`accessState = BLOCKED / PEDESTRIAN_ONLY` 跳过
4. **时间窗过滤**：`now ∈ [blockedFrom, blockedUntil)` 跳过；仅 `blockedFrom` 无 `blockedUntil` 视为永久封路
5. **方向语义**：BIDIRECTIONAL → 双向建边；FORWARD → 仅 from→to；REVERSE → 仅 to→from
6. **拥堵乘子**：`congestionLevel × 0.35 + (speedLimit < 10 ? 0.5 : 0)`
7. **边代价**：GPS 可用时用 haversine 米，否则 schematic 欧几里得；再乘以 edgeCostMultiplier

**寻路算法**：**Dijkstra**（`ParkRoutePlannerServiceImpl.shortestNodePath`），PriorityQueue + distances + previous，支持 edgePenalties。不可达抛 `PARK_ROUTE_NOT_FOUND`。

> 注：项目文档中有时称 A*，但实际代码使用 Dijkstra（A* 的退化形式，无启发函数）。

#### 3.4.3 路线安全闭环

**完整流程**（`RoadRouteValidateAdminService.validate`）：

```text
请求进入 → 生成 routeId（UUID）
  │
  ▼
1. 加载 VehicleRoutingProfile（从 VehicleEntity）
   含 widthCm/lengthCm/turningRadiusM/allowedRoadClasses/safetyBufferMeters
  │
  ▼
2. 起终点吸附（RouteEndpointSnapper.snapToNearestNode）
   ├─ 优先使用 accessNodeCode
   ├─ 否则按 haversine 找最近 ACTIVE 节点
   └─ 距离 > 50m → 返回 START_OFF_ROAD / END_OFF_ROAD
  │
  ▼
3. 路线规划
   ├─ 首选：ChainedRoadRouteService（本地路网 + 高德驾车）
   └─ 兜底：STRAIGHT_LINE（仅 allowStraightLine=true 时）
  │

4. 两层碰撞校验（RoadRouteCollisionValidator.applyValidation）
   ├─ 中心线碰撞：路线 polyline 与膨胀建筑 Polygon 相交
   ├─ 车辆包络膨胀：按 safetyBufferMeters 膨胀建筑 Polygon（GeoPolygonUtils.expandPolygon）
   ├─ 起终点位于建筑内 → startOrEndInsideBuilding
   └─ 离路距离 > 45m → offRoad
  │

5. 直线回退拦截
   STRAIGHT_LINE + allowStraightLine=false → 强制 invalid
  │
  ▼
```

#### 3.4.4 MAPF 多车路径规划

**机制**：时间桶 + 空间网格预留，避免多车冲突。

| 维度 | 配置 |
| --- | --- |
| 时间桶 | `bucket-ms=500`（500ms 一个桶） |
| 空间网格 | `zone-grid-size=4`（4×4 网格） |
| 预留范围 | `horizon-buckets=6`（向前看 6 个桶） |
| Redis Key（边） | `mapf:res:{parkId}:{from}>{to}:{bucket}` |
| Redis Key（节点） | `mapf:node:{parkId}:{node}:{bucket}` |
| TTL | `bucketMs × (horizon + 2)` |
| 冲突检测 | 检查反向边是否被其他车辆占用 |
| 失败回滚 | 释放所有已获取的 key |

#### 3.4.5 事件驱动与 Outbox 模式

**事件类型**（`DispatchEventType`）：`dispatch.task.created/assigned/manual-assigned/manual-pending/executing/success/failed/cancelled`、`dispatch.exception.open/resolved`、`dispatch.hub.arrival/departure`。

**Outbox 流程**：

```text
业务事务内
  │
  ├─ 1. outboxService.savePending(event) → 写入 t_dispatch_event_outbox（status=PENDING）
  │
  └─ 2. 注册 TransactionSynchronization.afterCommit 钩子
       │
       ▼
事务提交后
  │
  ├─ 3. rabbitTemplate.convertAndSend(DISPATCH_EXCHANGE, eventType, event)
  │     │
  │     ├─ 成功 → markPublished(eventId)
  │     └─ 失败 → markFailed(eventId, error)，retryCount++，nextRetryTime = now + min(retryCount×30s, 300s)
  │
  └─ 4. DispatchEventRetryScheduler 定期扫描 PENDING/FAILED 重投
```

**RabbitMQ 拓扑**：

- Exchange: `fsd.dispatch.exchange`（Topic, durable）
- Queues: `fsd.dispatch.audit.queue` / `fsd.dispatch.stream.queue` / `fsd.dispatch.webhook.queue`
- Binding: `dispatch.#` 绑定到三个队列
- 三个消费者：审计落库 / SSE 推送 / Webhook 投递

#### 3.4.6 充电策略

`FleetChargePolicy` 定义能量阈值：

| 判定 | 公式/阈值 |
| --- | --- |
| `isLowSoc(soc)` | soc < lowBatteryThreshold（默认 30） |
| `shouldReturnToCharge(soc)` | soc < minAssignableSoc（默认 20） |
| `isCriticalSoc(soc)` | soc < criticalThreshold（默认 10） |
| `isFullyCharged(soc)` | soc ≥ fullSoc（默认 100） |
| `isAssignable(soc)` | soc ≥ minAssignableSoc |

**充电队列优先级**：SOC 越低越优先；高峰时段 SOC > 50% 的车辆排在 SOC ≤ 50% 之后；只纳入 IDLE 状态车辆。

### 3.6 关键技术决策与设计思路

#### 3.6.1 为什么选 MyBatis-Plus 而不是 JPA？

| 考量 | 决策 |
| --- | --- |
| 复杂 SQL 控制 | MyBatis-Plus 保留 XML/LambdaQueryWrapper 灵活写复杂 SQL，JPA 复杂查询需 Specifications 或原生 SQL |
| 性能 | 编译期生成 SQL，无 Hibernate 反射开销 |
| 工程效率 | `BaseMapper<T>` 提供通用 CRUD，`LambdaQueryWrapper` 链式条件构造，开发效率高 |
| 与 MySQL JSON 字段配合 | MyBatis-Plus 类型处理器支持 JSON 字段映射 |

#### 3.6.2 为什么用 Outbox 模式而不是直接发 MQ？

**问题**：业务事务与消息发送的双写一致性问题。先发消息再提交事务，事务回滚时消息已发；先提交事务再发消息，进程崩溃时消息丢失。

**Outbox 方案**：

1. 业务事务内同步写 `t_dispatch_event_outbox`（同事务保证一致性）
2. 事务提交后异步投递 RabbitMQ
3. `DispatchEventRetryScheduler` 定期扫描失败记录重投

**收益**：事件不丢、事务一致、可重试、可观测（status/retryCount/lastError）。

#### 3.6.3 为什么维护三套坐标系？

| 坐标系 | 必要性 |
| --- | --- |
| Schematic | 园区示意场景下 A* 寻路需要拓扑图，像素坐标计算最快；早期无 GPS 数据时唯一可用 |
| GCJ-02 | 中国法规要求地图显示必须用 GCJ-02（火星坐标），高德/腾讯地图底图原生支持 |
| WGS-84 | 车辆 GPS 模块原始输出，必须转换后才能使用 |

**设计思路**：通过 `ParkGeoTransformService` 统一转换，存储层 GCJ-02 为主（显示零漂移），路网层 Schematic 为主（寻路高效），接入层 WGS-84 转 GCJ-02。

#### 3.6.4 为什么 Fleet 用 Redis 而不是 MySQL？

| 考量 | 决策 |
| --- | --- |
| 读写频率 | 车辆遥测每秒上报，前端 SSE 每秒推送，MySQL 扛不住 |
| 数据特性 | 运行态是临时数据（位置/速度/SOC），不需要持久化（轨迹点另存 MySQL） |
| TTL 管理 | Redis 7 天 TTL 自动清理，无需手动维护 |
| 批量读取 | `multiGet` 一次拉多辆车运行态，性能远超 MySQL |
| 分布式锁 | Redis 原子操作天然支持分布式锁（任务锁/服务位锁/MAPF 预约） |

#### 3.6.5 为什么用 FleetAdapter 适配器模式？

**问题**：SIM 仿真、REAL HTTP、VDA5050 MQTT 三种车辆链路接入方式差异巨大，但调度核心不希望感知差异。

**方案**：

```java
interface FleetAdapter {
    VehicleLinkMode supportedLinkMode();
    void ingestTelemetry(VehicleTelemetryRequest request);
}

@Component class SimulationFleetAdapter implements FleetAdapter { ... }
@Component class RealFleetAdapter implements FleetAdapter { ... }
@Component class Vda5050FleetAdapter implements FleetAdapter { ... }

@Component class FleetAdapterRegistry {
    private Map<VehicleLinkMode, FleetAdapter> adapters;
    FleetAdapter require(VehicleLinkMode mode) { ... }
}
```

**收益**：新增车辆链路只需实现接口 + 注册，调度核心零改动；运行时按 `vehicle.linkMode` 动态路由。

#### 3.6.6 为什么前端用 Composition API + Pinia setup 风格？

| 考量 | 决策 |
| --- | --- |
| 类型推断 | Composition API + `<script setup>` 完整 TypeScript 类型推断，Options API 需额外装饰器 |
| 逻辑复用 | Composable 函数（`useDeliveryGeo`/`useSseConnection`）跨页面复用，Mixins 已被官方不推荐 |
| Tree-shaking | setup 风格 store 便于按需引入，减小打包体积 |
| DevTools | Pinia DevTools 支持 time-travel 调试 |

#### 3.6.7 为什么 SSE 而不是 WebSocket？

| 考量 | 决策 |
| --- | --- |
| 业务特性 | 主要是服务端推送（事件流），客户端不需要频繁双向通信 |
| 协议简单 | SSE 基于 HTTP，无需额外协议升级，Nginx/Cloudflare 透传简单 |
| 自动重连 | 浏览器原生支持 EventSource 重连，仅需补充业务层重连逻辑 |
| 兼容性 | HTTP/1.1 即可，无需 HTTP/2 |
| Cloudflare 适配 | 看门狗机制规避 100s 超时，比 WebSocket 更稳定 |

#### 3.6.8 为什么 Capacitor 8 用远程 URL 模式？

**问题**：Android APK 发版周期长（应用商店审核），前端 bug 修复急需热更新。

**方案**：`capacitor.config.ts` 配置 `server.url = 'https://app.aplicity.online'`，APK 内 WebView 直接加载远程前端，前端更新无需重打包 APK。

**权衡**：

- 优势：热更新、版本统一、维护成本低
- 风险：依赖网络，离线不可用（通过 PWA Service Worker 缓解）

#### 3.6.9 为什么 Flyway 所有 ALTER 用 information_schema 判重？

**问题**：Docker 镜像构建缓存会导致 Flyway 迁移脚本更新后旧 jar 内嵌的 SQL 不更新；多环境部署时迁移可能重复执行。

**方案**：所有 `ALTER TABLE` 通过 `information_schema.COLUMNS` 判重，可重复执行（幂等）：

```sql
SET @sql := IF(
  (SELECT COUNT(*) FROM information_schema.COLUMNS
   WHERE TABLE_SCHEMA = DATABASE() AND TABLE_NAME = 't_xxx'
     AND COLUMN_NAME = 'new_field') = 0,
  'ALTER TABLE `t_xxx` ADD COLUMN `new_field` VARCHAR(64) DEFAULT NULL',
  'SELECT 1'
);
PREPARE stmt FROM @sql; EXECUTE stmt; DEALLOCATE PREPARE stmt;
```

**收益**：迁移失败可重试、多环境部署安全、回滚简单。

#### 3.6.10 为什么路线校验要两层碰撞 + 直线回退拦截？

**问题背景**：

1. 早期版本路线规划失败时回退到 `STRAIGHT_LINE`（起点到终点直线），车辆实际行驶会穿越建筑/水域
2. 空路线被误判为 `valid(0D)` 通过校验
3. 仅校验中心线，未考虑车辆自身宽度

**改进方案**：

1. 空路线改为 `invalid(false, false, 0D)`
2. 两层碰撞：中心线碰撞 + 车辆包络膨胀碰撞（按 `safetyBufferMeters` 膨胀建筑 Polygon）
3. `STRAIGHT_LINE` + `allowStraightLine=false` 强制 `invalid`，禁止直线兜底进入执行队列
4. 起终点位于建筑内 → `startOrEndInsideBuilding`
5. 离路距离 > 45m → `offRoad`

**收益**：路线安全性大幅提升，避免车辆穿越建筑/水域。
