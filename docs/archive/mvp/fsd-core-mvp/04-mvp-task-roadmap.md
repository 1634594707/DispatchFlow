# FSD-Core MVP 任务路线图

## 1. 文档目标

本文档用于将 `FSD-Core MVP` 的需求和设计拆解为可执行的阶段任务，明确每阶段目标、交付物和依赖关系，作为研发推进路线图。

## 2. 推进原则

- 先需求冻结，再进入实现
- 先跑通主链路，再扩展边角能力
- 先保证状态正确，再做性能优化
- 先保证可排障，再做体验优化

## 3. 阶段划分

MVP 建议拆为 6 个阶段。

### 阶段 0：需求与设计冻结

目标：

- 冻结 MVP 边界
- 冻结状态机和业务规则
- 冻结接口草案和模块边界

交付物：

- 功能需求文档
- 设计文档
- 接口定义清单
- 任务路线图

完成标准：

- 订单、调度、车辆、异常处理范围明确
- 关键业务规则不再口头变动

### 阶段 1：项目初始化

目标：

- 初始化后端工程结构
- 初始化前端管理端结构
- 补齐本地和基础环境

交付物：

- Maven 多模块项目骨架
- 前端项目骨架
- 本地 `MySQL / Redis / RabbitMQ` 启动方案
- 基础配置模板

任务项：

- 安装并配置 `Maven`
- 创建后端模块结构
- 初始化前端工程
- 配置统一代码规范和基础日志

### 阶段 2：核心数据模型与基础接口

目标：

- 建立核心表结构
- 实现订单、车辆、调度任务的基础读写

交付物：

- 数据库表设计与建表脚本
- 基础实体、DTO、VO
- 订单、车辆、调度任务基础查询接口

任务项：

- 建立 `t_order`
- 建立 `t_dispatch_task`
- 建立 `t_vehicle`
- 建立异常与操作日志表
- 实现基础 CRUD 与查询能力

### 阶段 3：调度主链路实现

目标：

- 跑通订单创建到派单成功的闭环

交付物：

- 创建订单接口
- 自动派单逻辑
- 手动派单逻辑
- 状态流转实现

任务项：

- 订单创建后进入待调度
- 调度模块生成任务
- 车辆候选集筛选
- 派单成功后更新订单、任务、车辆状态
- 编写主链路测试

### 阶段 4：车辆回传与异常处理

目标：

- 跑通执行态推进和异常待处理闭环

交付物：

- 车辆状态回传接口
- 异常记录与人工处理接口
- 调度监控摘要接口

任务项：

- 处理车辆在线/离线状态回传
- 处理任务执行推进
- 识别派单失败、车辆离线、超时等异常
- 异常进入人工处理池

### 阶段 5：管理端页面与联调

目标：

- 提供可用的调度后台页面

交付物：

- 订单管理页面
- 调度任务页面
- 车辆页面
- 异常处理页面

任务项：

- 完成接口契约对齐
- 完成页面筛选、列表、详情、操作能力
- 完成前后端联调

### 阶段 6：测试、验收与上线准备

目标：

- 验证 MVP 是否可交付

交付物：

- 测试报告
- 验收清单
- 部署方案
- 回滚预案

任务项：

- 核心流程回归测试
- 异常流程验证
- 日志与问题排查验证
- Docker 部署验证

## 4. 任务优先级

### P0

- 冻结需求范围
- 冻结状态机
- 初始化工程结构
- 建核心表
- 打通订单创建和派单主链路

### P1

- 车辆状态回传
- 异常处理闭环
- 管理端页面
- 前后端联调

### P2

- 基础看板摘要
- 缓存优化
- 事件链补偿增强
- 测试覆盖率提升

## 5. 建议迭代顺序

建议按以下顺序推进：

1. 文档冻结
2. 工程初始化
3. 数据模型落地
4. 后端主链路
5. 前端页面
6. 联调
7. 验收与部署

## 6. 角色分工建议

### 后端

负责：

- 数据模型
- 状态机实现
- 调度逻辑
- Redis 与 RabbitMQ 接入
- 日志与测试

### 前端

负责：

- 后台页面搭建
- 接口联调
- 列表、详情、操作流设计
- 错误态、空态、加载态处理

### 产品/业务

负责：

- 冻结业务规则
- 确认异常处理口径
- 确认状态迁移边界

### 测试

负责：

- 主链路用例
- 异常场景用例
- 回归验证

## 7. 风险清单

### 7.1 需求风险

- 业务规则未冻结导致反复返工
- 执行中任务的取消和改派口径不清

### 7.2 技术风险

- 过早拆服务导致开发复杂度上升
- Redis 与数据库状态不一致
- MQ 消费幂等未做好导致重复处理

### 7.3 联调风险

- 前后端枚举和时间格式不一致
- 状态回传字段口径不统一

## 8. 里程碑建议

### M1

- 文档冻结完成

### M2

- 后端主链路接口可用

### M3

- 车辆回传和异常处理闭环可用

### M4

- 管理端页面联调完成

### M5

- MVP 测试通过并具备部署条件

## 9. 路线图结论

MVP 推进应坚持“闭环优先、复杂度后置”的原则。只有先把订单、调度、车辆、异常处理的主链路跑通，后续优化才有意义。

本路线图可直接作为项目第一阶段的执行基线。
