# FSD-Core AI 项目上下文 Prompt

## 1. 文档目标

本文档提供一套可直接复用的 Prompt 模板，帮助 AI 在后续参与 `FSD-Core` 新功能推进时，快速理解项目背景、MVP 边界、技术约束和输出要求。

使用原则：

- 先整理需求，再写代码
- 高风险逻辑先设计，再实现
- 涉及状态机、Redis、RabbitMQ、事务边界时必须显式说明
- 先按 MVP 边界推进，不擅自扩展完整平台能力

## 2. 项目背景 Prompt

下面这段可作为 AI 的基础上下文：

```text
你正在参与 FSD-Core 项目。

项目背景：
FSD-Core 是商用无人配送车辆智能调度平台，当前阶段是 MVP 建设阶段。

MVP 目标：
1. 支持订单进入系统
2. 支持系统或人工完成车辆分配
3. 支持车辆执行状态回传
4. 支持调度员查看任务态势和异常
5. 支持异常任务进入人工处理流程
6. 支持核心链路日志追踪与问题排查

技术栈：
- Java 21 作为项目目标版本
- Spring Boot 3.x
- Spring Cloud Alibaba 按需引入
- MyBatis-Plus
- MySQL 8
- Redis 7
- RabbitMQ 3.x
- Vue 3 + TypeScript + Vite + Pinia + Element Plus

架构形态：
- MVP 阶段采用模块化单体，不做过早微服务拆分

核心模块：
- fsd-common
- fsd-order
- fsd-dispatch
- fsd-vehicle
- fsd-admin-api
- fsd-bootstrap

核心业务对象：
- Order
- DispatchTask
- Vehicle

核心业务闭环：
订单进入系统 -> 进入待调度 -> 自动或手动派单 -> 车辆执行 -> 状态回传 -> 异常处理 -> 完成或失败

关键状态：
订单状态：
- CREATED
- WAITING_DISPATCH
- DISPATCHED
- IN_PROGRESS
- COMPLETED
- CANCELLED
- FAILED

调度任务状态：
- PENDING
- ASSIGNING
- ASSIGNED
- EXECUTING
- SUCCESS
- FAILED
- CANCELLED
- MANUAL_PENDING

车辆状态：
在线状态：
- ONLINE
- OFFLINE
调度状态：
- IDLE
- BUSY
- UNAVAILABLE

关键规则：
1. 一个订单只对应一个主调度任务
2. MySQL 是最终业务真相源
3. Redis 不是最终真相源，只做缓存、幂等、短锁
4. RabbitMQ 只做异步事件传播，不承担业务真相
5. 自动派单失败后默认进入人工待处理
6. 执行中任务默认不允许改派
7. 离线车辆不得参与自动派单
8. 所有关键状态迁移必须通过服务层校验
9. Controller 不承载业务逻辑
10. DTO、VO、Entity 必须分离

你的工作方式：
1. 先整理需求
2. 再输出接口、模块、状态流转或数据设计
3. 最后才进入实现
4. 涉及 Redis、RabbitMQ、事务、幂等、锁、补偿时必须显式列出风险
5. 高风险逻辑必须给出测试建议
```

## 3. 新功能需求整理 Prompt

当你要让 AI 帮你梳理新功能时，直接使用：

```text
基于 FSD-Core MVP 项目背景，帮我整理下面这个新功能需求。

要求：
1. 先不要写代码
2. 先按 MVP 边界判断这个需求是否应该进入当前阶段
3. 输出内容必须包括：
   - 功能目标
   - 涉及角色
   - 核心流程
   - 状态变化
   - 接口需求
   - 数据模型影响
   - Redis 影响
   - RabbitMQ 影响
   - 风险点
   - 验收标准
4. 如果需求超出 MVP，请明确指出哪些内容应该延期
5. 如果涉及高风险逻辑，先输出设计草案，不要直接实现

新功能需求：
【在这里填入具体需求】
```

## 4. 新功能设计 Prompt

当需求已经明确，要让 AI 帮你出设计稿时，使用：

```text
基于 FSD-Core MVP 项目背景和既有模块边界，帮我为下面这个功能输出设计草案。

要求：
1. 不直接写代码
2. 按模块化单体架构设计
3. 明确放在哪个模块
4. 输出内容必须包括：
   - 涉及模块
   - 接口设计
   - DTO / VO 设计
   - 数据表或字段变更
   - 状态机影响
   - Redis 设计
   - RabbitMQ 事件设计
   - 事务边界
   - 幂等与并发控制
   - 测试建议
5. 不允许把业务逻辑塞进 Controller
6. 如果设计会破坏现有 MVP 边界，必须明确指出

功能名称：
【在这里填入功能名称】

功能描述：
【在这里填入功能描述】
```

```text

2. 明确会修改哪些模块和文件

   - 实现步骤
   - 代码改动范围
   - 风险点
   - 单元测试建议
   - 联调影响
```

## 7. 代码评审 Prompt

当你要让 AI 评审某次改动时，使用：

```text
基于 FSD-Core MVP 项目背景，对下面这次改动做结构化代码评审。

评审重点：
1. 功能正确性
2. 状态机是否被破坏
3. 事务边界是否清晰
4. Redis 使用是否安全
5. RabbitMQ 事件设计是否可靠
6. 幂等与并发控制是否充分
7. 是否违反模块边界
8. 是否缺少测试

输出要求：
1. 先列问题，按严重程度排序
2. 指出文件位置和风险
3. 最后再给简短总结

评审对象：
【在这里填入改动说明或代码范围】
```

当你要让 AI 协助联调或排障时，使用：

## 10. 使用建议

- 每次新功能推进，先用“新功能需求整理 Prompt”
- 需求明确后，再用“新功能设计 Prompt”
- 设计明确后，再进入“新功能实现 Prompt”
- 联调前优先用“接口设计 Prompt”
- 改动完成后用“代码评审 Prompt”

## 11. 结论

这份 Prompt 文档的目标不是替代设计，而是让 AI 在参与 `FSD-Core` 时始终带着统一的项目背景、边界意识和工程约束，减少上下文反复补充和方向偏移。
