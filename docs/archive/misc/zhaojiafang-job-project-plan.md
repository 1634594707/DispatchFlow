# 找家纺后端岗位定向项目规划

## 1. 目标定位

这个项目不是单纯做一个“会动的无人车大屏”，而是作为我应聘找家纺这类产业带无人配送/调度系统后端岗位的项目样本。

项目目标分为 3 层：

1. 证明我能独立完成一个有业务闭环的后端系统
2. 证明我理解园区无人车调度的真实业务，不是只会写 CRUD
3. 证明我能把后端能力落到前端演示、接口设计、状态流转、异常处理和部署方案上

## 2. 为什么选这个方向

根据公开资料，找家纺的无人配送场景不是单仓库 AGV，而是更贴近真实产业带的短驳配送系统，核心特点包括：

- 商户、工厂、仓库点位分散
- 高频小单、短距离配送
- 订单驱动车辆任务
- 车辆送完后需要回待命位或进入补能
- 需要调度、监控、异常处理、充电管理协同

这和当前项目 `DispatchFlow` 的园区模拟方向高度一致，适合作为定向求职作品继续打磨。

## 3. 项目最终要讲成什么

面试时，这个项目应该能被表述为：

“我做了一个面向园区/产业带短驳配送的无人车调度系统原型。系统包含下单、自动派车、订单执行、车辆位置追踪、停车充电联动、异常状态处理和大屏监控。我重点负责后端领域建模、接口设计、任务状态流转、仿真调度逻辑以及前后端联调。”

如果只能让面试官记住 4 个点，应该是：

- 订单创建后自动生成调度任务
- 车辆按状态机执行取货、送货、返航、充电
- 车辆监控大屏和移动下单端共用同一套后端接口
- 系统不仅有正常流程，也考虑了低电量、离线、人工介入等异常场景

## 4. 这个项目最重要的后端价值

### 4.1 业务建模

需要能讲清这些核心对象：

- 订单 `Order`
- 调度任务 `DispatchTask`
- 车辆 `Vehicle`
- 站点 `Station`
- 道路节点 `RoadNode`
- 停车充电位 `ParkingSpot`
- 车辆运行态 `RuntimeStage`

### 4.2 状态流转

项目的亮点不是表结构，而是状态流转是否合理。

当前应重点强调这条链路：

`待分配 -> 前往取货 -> 装货 -> 前往送货 -> 卸货 -> 电量评估 -> 回待命/去充电 -> 充电完成 -> 回待命`

异常链路至少要覆盖：

- 车辆离线
- 低电量无法继续执行
- 无可用车辆，任务进入人工介入
- 任务失败后保留可追踪状态

### 4.3 接口设计

需要体现“不是写死页面数据”，而是面向多端复用设计接口。

当前项目应保留并继续扩展这几类接口：

- 园区布局接口
- 站点列表接口
- 车辆快照接口
- 订单快照接口
- 移动端下单接口
- 后续可扩展的任务详情、告警接口、充电状态接口

### 4.4 工程能力

这个项目除了业务，还要体现工程化能力：

- Spring Boot 多模块拆分
- 领域服务分层
- 集成测试
- 本地开发与部署说明
- 前后端联调闭环

## 5. 当前项目已经具备的价值

截至当前版本，这个项目已经有一些适合面试展示的基础：

- 有前端管理端和车辆监控大屏
- 有后端多模块结构
- 有园区仿真与车辆轨迹展示
- 已有简化下单接口 `POST /api/admin/park/orders`
- 已支持订单驱动车辆执行任务
- 已支持送货结束后按电量回待命位或进入充电
- 已把停车位和充电位合并建模
- 已有移动端简化下单页

这说明项目已经从“展示页面”进入“可讲业务闭环”的阶段。

## 6. 当前还缺什么

为了更适合找家纺后端岗位，这个项目还需要继续补 4 类能力。

### 6.1 后端业务深度还不够

还需要补：

- 任务详情接口
- 车辆详情接口
- 充电状态接口
- 告警/异常事件接口
- 调度日志或任务轨迹查询接口

### 6.2 数据持久化表达还不够强

如果现在主要依赖仿真内存状态，面试时说服力不够。

建议逐步补：

- 订单持久化
- 调度任务持久化
- 关键状态变更日志持久化
- 车辆告警记录持久化

不要求一步做成完整生产系统，但要能证明设计上考虑过“从仿真到真实系统”的迁移路径。

### 6.3 异常处理可视化还不够

需要让面试官看到“系统不仅能跑，也知道什么时候跑不动”。

建议补：

- 低电量告警
- 离线告警
- 任务超时告警
- 人工介入列表

### 6.4 调度策略还偏简单

如果只是“选一台车就去送”，会显得过于演示化。

后续建议能讲清调度策略至少考虑：

- 是否空闲
- 是否在线
- 当前电量是否足够完成任务
- 距离取货点远近
- 是否已有未完成任务

## 7. 对应找家纺岗位，项目最该强化的模块

如果目的是找工作，优先级不要平均分配，建议这样排：

### P0：必须完成

- 订单创建到任务执行的完整闭环
- 车辆送货后电量评估与回待命/充电
- 大屏实时监控
- 移动端下单联动后端
- README 和本地运行文档清晰

### P1：强烈建议完成

- 任务详情页/接口
- 车辆详情页/接口
- 告警列表/接口
- 调度策略说明文档
- 集成测试补强

### P2：加分项

- WebSocket 实时推送替代轮询
- 调度日志检索
- 权限与登录

## 8. 面试导向的页面与能力清单

这个项目最终最好能演示 5 个页面/视角：

### 8.1 大屏监控

展示内容：

- 园区底图
- 车辆实时位置
- 订单链路
- 停车充电观察层
- 低电量、充电、忙碌、离线状态

### 8.2 移动端下单页

展示内容：

- 选择取货点/送货点
- 提交订单
- 订单创建成功
- 在同页追踪车辆和订单状态

### 8.3 订单列表/详情

展示内容：

- 订单当前阶段
- 分配车辆
- 对应任务
- 更新时间
- 是否异常

### 8.4 车辆列表/详情

展示内容：

- 在线状态
- 电量
- 当前任务
- 当前目标点
- 是否正在充电

### 8.5 告警或人工介入页

展示内容：

- 低电量
- 离线
- 无可用车辆
- 手动待处理任务

## 9. 应聘时该怎么讲这个项目

### 9.1 项目背景

“我参考了产业带无人配送场景，做了一个园区级无人车调度系统原型，重点模拟订单驱动调度、车辆执行任务、停车充电协同和监控联动。”

### 9.2 我的职责

“我主要负责后端建模、接口设计、调度状态机、仿真逻辑、测试以及前后端联调。”

### 9.3 技术亮点

“项目不是静态页面，而是通过订单创建驱动后端生成任务，再让车辆按状态机执行。送货完成后系统会自动判断电量，决定返回待命位还是进入充电位，同时前端监控大屏和移动端共享同一套接口。”

### 9.4 可继续优化的点

“如果继续做生产化，我会进一步补充实时推送、告警中心、权限体系、任务持久化和更真实的调度策略。”

## 11. 结论

这个项目最适合被定位成：

“面向产业带无人配送场景的调度系统后端样板项目”

接下来开发时，原则不是“功能越多越好”，而是：

- 每个功能都能服务于后端岗位面试表达
- 每个页面背后都要有清晰的业务模型和接口
- 每个异常状态都要能说明我理解真实调度系统

如果后续继续投入，优先把“任务详情、车辆详情、告警中心、测试补强”做好，这会比继续堆 UI 更能提高求职含金量。
