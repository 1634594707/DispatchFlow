# 真实地图接入 — 竞品调研与选型评估

> V3 · M1–M4 · 最后更新：2026-05-31  
> 状态：**调研完成，待 PoC 选型**

---

## 1. 背景：DispatchFlow 现状

| 维度 | 当前实现 | 局限 |
|------|----------|------|
| 前端地图 | Leaflet + `L.CRS.Simple` + `park-map.svg` | 园区像素坐标，非真实经纬度 |
| 坐标数据 | `t_park.map_width/height`、站点 `x/y` | 无 `center_lat/lng`、无 GCJ/WGS 字段 |
| 车辆遥测 | `VehicleReportRequest.latitude/longitude` 已预留 | REAL/VDA5050 上报未统一落库与展示 |
| 路径规划 | 自建路网图 A*（`ParkRoutePlannerServiceImpl`） | 与公开道路网无关，适合室内/封闭园区 |

**V3 目标**：在保留「园区路网调度内核」的前提下，接入**真实地理底图**，支持多园区地理总览、REAL 车队定位、地理围栏与（可选）公开道路路径辅助。

## 2. 同类产品地图方案

### 2.1 室内 / 封闭园区 AMR·AGV 调度（与 DispatchFlow 最接近）

| 产品 | 地图来源 | 可视化 | 说明 |
|------|----------|--------|------|
| **仙工智能 M4 / Meta-Map** | 自研：CAD + SLAM 栅格/矢量 | 2D/3D 自研引擎 | 不依赖高德/百度；地图=机器人可执行层 |
| **极智嘉 / 海康 / 快仓** | 厂商 SLAM / 二维码 / 反光板 | 自研或 Web 画布 | FMS 底图与调度图同构，偏工厂 |
| **Open-RMF / 开源 FMS** | 用户导入 floor plan | RViz / 自研 Web | 语义地图 + 交通调度，无互联网地图 |

**结论**：封闭场景 **FMS 核心仍用自建路网**；互联网地图仅作「运维大屏 / 多园区 GIS 壳层」，不是替换 A* 调度图。

### 2.2 室外 L4 无人配送（新石器 / 白犀牛 / 九识等）

| 能力 | 典型方案 |
|------|----------|
| 定位 | 北斗/GPS + SLAM，厘米级惯导补盲 |
| 地图演进 | 早期高精地图 → **2025+ 无图化**（BEV + 普通导航地图） |
| 公开地图角色 | **普通导航地图**作道路语义参考；核心感知/规划自研 |
| 团队背景 | 多来自百度 Apollo / 京东物流自动驾驶 |

**结论**：公开地图（高德/百度）是**运营与导航辅助**，不是调度系统主地图；DispatchFlow 若接 REAL 室外车，需 WGS84/GCJ02 管道，但**不应**用 Web 地图 API 替代园区路网。

### 2.3 物流调度 / 车队平台（业务层）

| 产品 / 能力 | 地图侧 | 调度侧 |
|-------------|--------|--------|
| **高德物流服务 API** | 无强制前端 SDK | 矩阵算路、N-1 最优车、排线调度（200–2000 点） |
| **百度物流 / 货车批量算路** | JSAPI GL / Web 服务 | 限行、限高限重、批量 matrix |
| **百度具身智能 / 园区导航** | 私有化园区地图 + 电子围栏 | 面向具身机器人/无人车的园区通行 |
| **顺丰 / 京东 / 货拉拉** | 多接入高德物流能力 | 末端排线 + 轨迹监控 |

**结论**：若 V3 要做「跨园区 REAL 路径估算 / 派单距离矩阵」，**高德物流 API** 与业务最贴近；百度在货车四限、园区具身方案上文档更全。

### 2.4 智慧园区 / 数字孪生平台

| 平台 | 做法 |
|------|------|
| 腾讯微瓴等 | 腾讯地图 LBS + 室内蓝牙/UWB；室外/室内一体导航 |
| 百度车联网 | 车机版地图 + ADASIS 输出，偏车载导航而非 FMS |

## 3. 高德 vs 百度 — 对比（DispatchFlow 视角）

| 维度 | 高德地图 | 百度地图 | 对 DispatchFlow 的建议 |
|------|----------|----------|------------------------|
| **坐标系** | GCJ-02（火星） | BD-09（需转换） | 优先 GCJ-02，与多数车载 GPS 一致 |
| **Web 前端** | JS API 2.0 + `@amap/amap-jsapi-loader`（可 npm 动态加载） | JSAPI GL / `vue-bmap-gl` / `@baidumap/mapv-three` | Vue3 工程更顺：**高德 loader** |
| **物流后端 API** | 物流矩阵、排线、插单（`tsapi.amap.com`） | 货车 matrix、路线规划 | 派车矩阵：**高德**；货车四限：**百度** |
| **车队监控** | 轨迹、Marker 聚合、自定义 overlay | 鹰眼轨迹 SDK、GL 轨迹线 | 大屏监控两者均可；高德聚合文档多 |
| **园区 / 具身** | 室内定位、IoT 合作方案 | **具身智能园区导航**、私有化上图 | 若卖园区私有化：**百度方案可对标** |
| **合规** | Key + `securityJsCode`、域名白名单 | AK + 白名单 | 均需企业主体认证；Key 放 env |
| **费用** | 日配额 + 物流 API 按次 | 类似 | PoC 免费额度够用；生产按 DAU/调用量 |

### 3.1 推荐策略：**双轨地图架构**

```text
┌─────────────────────────────────────────────────────────┐
│  DispatchFlow 地图层                                     │
├──────────────────────┬──────────────────────────────────┤
│  L1 调度图（保留）     │  L2 地理图（V3 新增）              │
│  Leaflet CRS.Simple  │  高德 or 百度 JS API              │
│  路网节点 x/y        │  WGS84/GCJ02 经纬度               │
│  A* / MAPF / 仿真    │  底图 / 围栏 / 多园区总览 / 轨迹   │
└──────────────────────┴──────────────────────────────────┘
         ↑ 仿射变换 / 控制点标定（每园区 3+ 对点）
```

- **L1 不删**：现有派车、仿真、VDA5050 联调仍走园区路网。
- **L2 新增**：监控大屏、运维 APP、多园区中心可切换「地理模式」。
- **MapProvider 抽象**：`AmapProvider | BaiduProvider | OfflineProvider`，配置项切换，避免写死。

### 3.2 首选与备选

| 优先级 | 选型 | 理由 |
|--------|------|------|
| **PoC 首选** | 高德 JS API 2.0 + 物流矩阵（可选） | GCJ-02、Vue3 生态、与调度矩阵能力对齐 |
| **并行评估** | 百度 JSAPI GL | 同屏对比渲染、围栏 UX、具身园区方案 |
| **保留** | Leaflet + 自定义 tile | 离线/内网园区、无 API Key 演示环境 |

## 5. PoC 验收标准

| # | 场景 | 预期 |
|---|------|------|
| 1 | 同一园区切换「调度图 / 地理图」 | 车辆 marker 位置一致（误差 < 5m） |
| 2 | REAL 车上报 GCJ-02 | 大屏地理模式实时刷新 |
| 3 | 3 控制点标定 | 保存后重启仍有效 |
| 4 | 无 Key 环境 | 回退 Leaflet 调度图，不白屏 |
| 5 | 高德 vs 百度 PoC 报告 | 加载耗时、内存、轨迹 100 点 FPS |

## 6. 风险与边界

| 风险 | 缓解 |
|------|------|
| 坐标系混用（BD-09 / GCJ / WGS） | 入库统一 GCJ-02，前端按 provider 转换 |
| 室内 GPS 漂移 | 地理模式标注「仅供参考」；调度仍以 L1 路网为准 |
| API 配额 / 密钥泄露 | 后端代理敏感 API；前端仅 Key + 域名限制 |
| 与 MAPF 路线不一致 | 地理路径仅展示/估算，不参与派车决策（V3） |

---

## 7. 结论

1. **竞品普遍「调度图自研 + 地理图可选」**，不应直接用互联网地图替换现有路网调度。
2. **V3 选型结论：高德 JS API 2.0（GCJ-02）** — 坐标系、Vue3 集成、物流 API 与 DispatchFlow 派单场景最匹配；百度不纳入 V3 实现。
3. **双轨架构**：L1 Leaflet 调度图保留；L2 高德地理底图（M1 PoC 已落地，M2 接入监控大屏）。
