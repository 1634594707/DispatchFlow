# DispatchFlow 前端站点与后端逻辑审查路线图

> 审查日期：2026 年 7 月 18 日  
> 范围：前端调度工作台、站点管理、地图展示、道路寻路、车辆派单和多车路径规划。  
> 本次只做审查和路线图整理，**不写代码**。

## 0. 总判断

当前界面最突出的问题不是“地图颜色不好看”，而是用户无法快速判断：

- 这个站点到底是现实地图站点，还是园区示意站点？
- 这个点是取货、送货、充电、待命，还是仅用于展示？
- 车辆路线是地图服务返回的真实道路，还是前端/后端生成的示意折线？
- 站点坐标、接入道路节点和服务位是否指向同一个物理位置？
- 当前显示的任务、车辆、充电和异常是否来自同一园区、同一地图版本和同一时间点？

建议把产品目标从“地图上显示很多东西”调整为：

> **让调度员在 5 秒内判断任务是否可执行、车辆能否到达、站点是否可服务、路线为什么这样走。**

## 1. 截图界面审查结论

### 1.1 主界面地图问题

- [ ] **地图语义不清**：截图中同时出现真实底图、绿色/蓝色区域边界、青色路线、站点白色标签、车辆图标和 L0/L1/L2 图层控件，但没有明确说明每层代表什么。
- [ ] **线路看起来像穿过街区**：青色路线从左下向右上穿过大面积区域，视觉上不像沿道路中心线行驶；需要通过真实道路折线和建筑物碰撞校验确认，而不是仅凭颜色判断。
- [ ] **站点标签互相覆盖**：多个站点标签集中在地图下半部，文字框压住路线和车辆，无法快速看出站点顺序。
- [ ] **站点与服务点概念混在一起**：`PICK`、`CHG`、`PARK` 等编码同时出现在地图上，但没有把“业务站点”“充电服务位”“停车位”“道路接入点”分开显示。
- [ ] **图例信息不足**：底部图例只解释车辆、站点、配送线、回充走廊，未解释区域 Polygon、建筑、不可通行区、封路、门禁和站点服务方向。
- [ ] **地图缺少状态解释**：没有明显显示地图数据时间、地图版本、路网健康状态、坐标系和当前路线来源。
- [ ] **地图比例与运营距离不一致风险**：右下角出现“1.57 × 0.47 km”一类范围信息，但需要确认它是视口尺寸、园区边界尺寸还是路线统计，避免调度员误读。
- [ ] **地图层级切换不符合任务流**：L0/L1/L2 更像技术图层名，不像调度员可理解的“园区总览 / 试点区域 / 服务位详情”。

### 1.2 主界面下方“页面分工”问题

截图中的“上线后的页面分工”属于解释性宣传区，不属于调度员当前决策所需的信息：

- [ ] 从主工作台下方移除“工作台只做快决策 / 报表只做复盘 / 设施只做建模”这类页面分工说明。
- [ ] 不要在主界面继续解释导航菜单职责；菜单和页面标题已经承担了这项职责。
- [ ] 删除后将空间用于“路线健康 / 当前阻塞 / 站点不可达 / 需要人工介入”的实时信息。
- [ ] 如果产品仍需要说明页面职责，放入帮助中心、首次使用引导或系统说明页，不放在主工作台首屏。

### 1.3 主界面信息架构建议

主工作台建议只保留四块核心信息：

1. **任务队列**：待派、执行中、超时、人工介入。
2. **态势地图**：车辆、任务路线、站点状态、封路和充电状态。
3. **资源与风险**：可派车辆、低电量、充电排队、路网异常。
4. **下一步动作**：自动派车、重规划、关闭道路、处理异常。

其余内容应通过侧边抽屉、详情页或二级页面展开，避免首屏同时承担运营、报表、基础设施说明和产品宣讲。

## 2. “站点很奇怪”应如何调研

### 2.1 先定义站点，而不是先改图标

在调研前先把“站点”拆成四个对象：

- [ ] **业务站点**：取货点、送货点、快递接驳点。
- [ ] **服务位**：车辆真正停靠、装卸或等待的位置。
- [ ] **接入道路节点**：服务位连接到哪一个道路节点或道路边。
- [ ] **资源位**：充电桩、停车位、待命位、换电柜等具体资源。

一个业务站点可以有多个服务位，一个服务位只能绑定清晰的接入道路和服务方向。不要再用一个经纬度同时代表建筑、门口、车辆停靠点和路线终点。

### 2.2 站点调研问题清单

- [ ] 站点名称是否对应现场真实门店、仓库或门牌？
- [ ] 站点类型是否与实际业务一致：取货、送货、充电、待命、快递？
- [ ] 坐标是否落在建筑物内部、道路中心、道路边缘还是服务位？
- [ ] 车辆从哪个方向进入？是否允许反向进入？
- [ ] 到站后是路边停靠、进入院内、进入月台还是进入充电区？
- [ ] 是否存在多个门、多个装卸口或不同车辆尺寸对应的服务位？
- [ ] 是否有人车混行、门禁、限高、限宽、临时封路或高峰禁停？
- [ ] 站点容量是“同时停靠车辆数”还是“单位时间处理任务数”？
- [ ] 站点是否有服务时长、排队时长和预约规则？
- [ ] 站点是否仅用于调度，不应出现在移动端下单或主地图？
- [ ] 站点坐标、接入节点、服务方向和地图版本是否同时有效？

### 2.3 建议的调研方式

#### A. 桌面地图核验

- [ ] 使用高德/天地图/现场已有 GIS 底图核验道路、建筑、门口和园区边界。
- [ ] 使用卫星图或现场 CAD/园区图确认建筑轮廓，不把地图上的兴趣点直接当成可停靠点。
- [ ] 记录每个点的 WGS84/GCJ-02 来源，不要混写坐标。
- [ ] 对每个站点截图标注“建筑轮廓、道路中心线、车辆服务位、接入节点”。

#### B. 现场走访

- [ ] 让司机或现场管理员从入口实际走到每个取货/送货点。
- [ ] 记录车辆能否转弯、掉头、会车、等待和离开。
- [ ] 记录门禁开启时段、人工指挥点、临时占道和高峰拥堵。
- [ ] 使用手机 GPS 只做辅助采集，最终点位需要落到道路和服务位模型。

#### C. 小样本任务回放

- [ ] 选 10 条真实任务，覆盖近距离、跨区域、充电中断、站点不可达和封路场景。
- [ ] 对比“系统路线、地图服务路线、司机实际路线”。
- [ ] 记录路线偏差、额外距离、等待时间、到站位置和任务完成状态。
- [ ] 把每个异常归因到坐标、路网、站点、派单、车辆状态或前端展示，而不是只记为“地图不准”。

## 3. 前端站点相关问题

### 3.1 真实站点和示意站点使用两套分类规则

**相关文件：** `front/src/maps/stationLayers.ts`

当前前端使用以下隐式规则区分站点：

- `area === 'ZJF'` 或编码以 `ZJF-` 开头，判定为真实配送站点。
- 非上述条件的站点，判定为园区示意站点。
- `ZJF-IDLE-01` 和 `ZJF-CHG-*` 被当成仅调度/回充站点。
- 移动端、工作台和示意地图对站点使用不同过滤函数。

风险：

- [ ] 站点是否真实取决于编码前缀，而不是数据库中的地图版本、站点用途和几何数据。
- [ ] 新增站点如果编码不符合约定，可能被错误显示或被过滤掉。
- [ ] `GENERAL` 区域会同时出现在多个模式，容易造成重复显示。
- [ ] “站点不可下单”和“站点不可调度”由编码规则表达，缺少显式能力字段。
- [ ] 真实地图和示意地图的站点可能使用同一个页面组件，但坐标含义不同。

### 3.2 站点表单允许多套坐标并存

**相关文件：** `front/src/views/infrastructure/StationList.vue`

当前表单同时维护：

- 园区屏幕/示意坐标 `coordX`、`coordY`。
- GCJ-02 经纬度 `coordLng`、`coordLat`。
- 接入道路节点、服务方向等字段在响应模型中存在，但站点编辑表单没有形成完整的服务位配置流程。

风险：

- [ ] 用户可以填写 X/Y 与经纬度，但没有明确哪一个是权威坐标。
- [ ] 地图点选会自动写入两套坐标，但未明确转换误差和坐标版本。
- [ ] 站点可以存在坐标，却没有道路接入关系，最终只能被“吸附到最近节点”。
- [ ] 站点可显示不等于车辆可到达；表格中的“可达”不应只依据坐标是否存在。
- [ ] 充电点、待命点和业务站点共用站点模型，容量和服务行为容易混淆。

### 3.3 地图上应该显示服务位，而不是只显示站点

- [ ] 站点图标显示业务类型。
- [ ] 服务位显示车辆真实停靠点。
- [ ] 接入节点显示道路连接关系。
- [ ] 充电桩显示具体桩位和排队状态。
- [ ] 建筑中心只用于识别建筑，不作为车辆导航终点。
- [ ] 点击站点时展示“站点 → 服务位 → 接入道路 → 当前可达性”的链路。

## 4. 后端逻辑审查发现

### 4.1 高风险：路线起终点没有真正吸附到道路

**相关文件：** `back/fsd-dispatch/src/main/java/com/fsd/dispatch/service/impl/ParkRoutePlannerServiceImpl.java`

当前寻路会先找到最近道路节点，但 `buildRouteFromNodePath()` 仍把调用方传入的原始 `startX/startY` 和 `endX/endY` 直接放入路线首尾，中间才放入道路节点。

这会产生一个关键问题：

> 即使中间路径沿道路，起点到最近节点、最后一个节点到终点仍可能是没有经过道路和障碍物检查的直线段。

需要重点验证取货点、送货点、充电点是否因此出现“最后一段穿建筑”或“停到道路中心”的现象。

### 4.2 高风险：空路网时 `isReachable()` 返回 true

`ParkRoutePlannerServiceImpl.isReachable()` 在 `graph.isEmpty()` 时直接返回 `true`，但 `buildRoute()` 在空路网时会抛出 `PARK_ROAD_NETWORK_EMPTY`。

这会造成校验与实际派单结果不一致：

- [ ] 可达性接口说“可达”。
- [ ] 真正生成路线时说“路网为空”。
- [ ] 前端可能提前放行站点或任务，直到派单时才失败。

空路网应被视为不可达或“地图未配置”，不能返回成功。

### 4.3 高风险：路线规划失败后的兜底语义不统一

`MapfRoutePlannerService` 在多车预约失败后，会调用普通 `buildRoute()` 生成未预约的路线并返回 `reserved=false`。

这不是一定错误，但必须明确消费方行为：

- [ ] 如果 `reserved=false` 仍然下发车辆，可能造成多车冲突。
- [ ] 如果只是预览路线，前端必须明显标记“未锁定，不代表可执行”。
- [ ] 如果预约失败是因为冲突，应返回“候选路线”或“等待重试”，而不是看起来像成功的路线。
- [ ] 应区分“找不到路”和“找到路但无法获得时空资源”。

### 4.4 中风险：示意路网回退仍存在直线来源

`LocalPilotRoadGraphService` 在 OSM 子图不连通、手工走廊找不到或路径点不足时返回 `RoadRouteSource.STRAIGHT_LINE` 的空结果；上层服务也保留了对空/无效路线的直线构造函数。

虽然当前链式服务日志写明“不再直线兜底”，但需要通过调用链和接口响应确认：

- [ ] 所有生产派单入口都经过 `ChainedRoadRouteService`。
- [ ] 所有路线响应都带 `source`、`invalid` 和不可达原因。
- [ ] 任意 `STRAIGHT_LINE` 结果不能进入车辆执行队列。
- [ ] 直线只允许用于“无路线的诊断提示”，不能用于车辆动画和真实派单。

### 4.5 中风险：数据库路网与 YAML/手工走廊可能产生多套事实源

`ParkRoutePlannerServiceImpl.resolveGraph()` 在数据库有节点时使用数据库，否则回退到 YAML；`LocalPilotRoadGraphService` 还保留手工走廊和固定站点锚点。

风险：

- [ ] 同一个园区可能同时存在数据库路网、YAML 路网、OSM 子图、手工走廊四套来源。
- [ ] 一套来源被更新，另一套来源没有同步，导致前端路线与后端路线不一致。
- [ ] 固定锚点与数据库站点坐标可能漂移。
- [ ] 地图版本表目前更像发布元数据，尚需验证路线查询是否强制绑定激活版本。

### 4.6 中风险：坐标距离计算存在模式混用

`ParkRoadGraph.NodeView.distanceTo()` 在有经纬度时使用 Haversine，没有经纬度时使用示意 X/Y 欧氏距离。路线首尾点可能只有示意坐标，中间节点可能带 GPS 坐标。

需要避免：

- [ ] 一条路线前后使用像素单位，中间使用米，导致距离和 ETA 不可信。
- [ ] 最近节点选择使用 X/Y，而路线展示使用经纬度，导致点位吸附错位。
- [ ] 前端将 GPS 点直接投到 SVG 像素坐标，但未使用同一个投影参数。

### 4.7 中风险：路段状态有过滤，但车辆约束仍需核验

数据库图目前已经考虑 `ACTIVE`、`BLOCKED`、`PEDESTRIAN_ONLY` 和临时封路时间窗，也有 `direction` 方向字段处理。

但仍需通过代码和测试确认：

- [ ] 车辆宽度、长度、最小转弯半径是否真正参与选路。
- [ ] 允许车辆类型是否真正过滤道路和服务位。
- [ ] 限高、限宽、消防通道和人车混行是否有实际约束。
- [ ] 拥堵倍率是否与限速、等待时间和路口占用重复计算。
- [ ] 仅禁用路段是否会导致已在路上的车辆突然失去当前位置和后续路线。

### 4.8 中风险：站点服务位预留需要并发一致性验证

后端已经有 `StationServicePosition` 和 `StationServicePositionReservation` 实体，也有站点服务位管理入口。

需要重点检查：

- [ ] 同一个服务位被两个车辆同时预留时是否有数据库唯一约束或原子更新。
- [ ] `reservedUntil` 到期后是否自动释放，还是只能依赖定时任务。
- [ ] 车辆取消任务、断线、异常和充电超时是否都会释放服务位。
- [ ] 服务位状态与站点状态、任务状态、车辆状态是否会出现互相矛盾。
- [ ] 预约成功但路线随后重规划时，旧服务位是否释放。

### 4.9 中风险：派车条件与任务状态需要做状态机审计

项目包含自动派车、手动派车、紧急插队、暂停派单和异常处理多个入口。

需要逐条核对：

- [ ] 暂停派单时，自动派车、批量派车、紧急插队是否全部受控。
- [ ] 任务从待派到已派、执行中、到站、装卸、完成、失败是否只允许合法迁移。
- [ ] 手动派车是否绕过电量、车型、路网和服务位校验。
- [ ] 同一任务重复点击派车是否幂等。
- [ ] 自动派车与手动派车并发时是否存在重复占用车辆。
- [ ] SSE/重试/消息重复消费是否会重复推进状态或重复创建车辆命令。

### 4.10 低风险但影响运维：错误信息编码和可解释性

后端多个异常消息和注释仍出现乱码形式。即使不影响编译，也会直接影响调度员排障。

- [ ] 统一异常码和中文展示文案。
- [ ] 前端不要只显示“路线失败”“派车失败”。
- [ ] 至少返回：地图版本、路线来源、起点服务位、终点服务位、失败阶段和失败原因。

## 5. 路线图

### Phase 0：冻结问题范围和验收口径

- [ ] 明确主工作台只服务调度决策，不在底部展示“页面分工”说明。
- [ ] 明确站点、服务位、接入节点、充电桩、停车位五个对象的定义。
- [ ] 明确唯一权威坐标和坐标转换链路。
- [ ] 选定 10 条真实任务作为固定回放集。
- [ ] 规定任何 `STRAIGHT_LINE` 路线不得进入真实车辆执行。

### Phase 1：站点数据调研和清洗

- [ ] 建立站点台账：编码、名称、类型、区域、建筑、门口、服务位、接入道路、方向、容量、服务时长。
- [ ] 逐点核验真实地图和现场路线。
- [ ] 清理重复站点、过期站点、建筑中心点和只用于示意的站点。
- [ ] 把 `ZJF-*`、`A1/B1` 等编码规则降级为兼容逻辑，不再作为唯一业务判断。
- [ ] 为站点补充显式能力：可下单、可取货、可送货、可充电、可待命、仅调度。
- [ ] 为每个业务站点配置至少一个服务位和接入节点。

### Phase 2：地图和路网统一

- [ ] 统一数据库路网、OSM 子图、手工走廊和 YAML 回退的事实来源。
- [ ] 给道路补齐真实几何、宽度、方向、限速、车辆类型和临时封路属性。
- [ ] 给建筑、围墙、河道、门禁和不可通行区域建立几何对象。
- [ ] 建立激活地图版本，并让路线规划、站点可达性和前端展示绑定同一版本。
- [ ] 统一所有路线的距离单位为米，统一 ETA 的速度和等待时间来源。

### Phase 3：路线安全和站点可达性

- [ ] 起点和终点先吸附到服务位/道路接入点，再进入图搜索。
- [ ] 对车辆包络和建筑物膨胀区做全路径碰撞检查。
- [ ] 空路网、无路径、站点未吸附、服务位占用和车辆不适配分别返回不同错误。
- [ ] 禁止无效路线、直线路线和未通过碰撞检查的路线进入执行队列。
- [ ] 增加路线来源、地图版本、节点序列、服务位和校验结果到路线响应。
