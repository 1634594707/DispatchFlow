# DispatchFlow 无人车真实道路配送参数补齐清单

> 整理日期：2026 年 7 月 18 日  
> 目的：说明为什么无人车有时不沿真实道路行驶、可能横穿建筑，以及在写代码前必须补齐的输入参数、数据关系和验收规则。  
> 本文只做审查和参数清单，**本次不修改代码**。

## 1. 先给结论

无人车要保证沿正常道路行走，不能只传入“起点坐标 + 终点坐标”。完整输入至少要包含：

1. 起点/终点的坐标系和坐标来源。
2. 起点/终点对应的业务站点、服务位和道路接入节点。
3. 道路图节点、路段几何、方向、宽度、限速和通行状态。
4. 建筑物、围墙、河道、步行区、门禁和其他不可通行区域。
5. 车辆尺寸、车型、最小转弯半径、安全缓冲和允许道路类型。
6. 路线规划结果的节点序列、路段序列、真实折线、路线来源和地图版本。
7. 路线碰撞校验、站点可达性校验和失败原因。
8. 车辆沿真实折线推进时的速度、时间戳、方向角和偏航处理。

缺少其中任意一组，系统就可能出现以下现象：

- 起点或终点在建筑物中心，最后一段直线穿过建筑。
- 路线规划失败后，车辆退回示意图坐标路线。
- 车辆中心点没有碰到建筑，但车身已经擦进建筑或道路外。
- 线路显示沿道路，车辆动画却使用另一套 X/Y 坐标移动。
- 站点能在地图上显示，但没有可停靠服务位，车辆只能“开到点中心”。
- 地图版本、道路版本和车辆轨迹版本不一致，导致路线看似正常但实际穿越旧障碍物。

## 2. 当前项目已经有的能力

项目不是完全没有路线能力，当前已经存在以下模块或字段：

- `RoadRouteService`、`ChainedRoadRouteService`：支持高德路线和本地路网链路。
- `RoadRouteResult`：包含折线、距离、路线来源、无效标记、穿建筑标记、穿河道标记和离路距离。
- `RoadRouteCollisionValidator`：检查建筑、河道和路线离道路距离。
- `RoadRouteFollower`：按真实折线弧长推进地理位置。
- `ParkRoadGraph`：根据道路节点和路段构建图，并处理部分道路状态、方向和临时封路。
- `VehicleRoutingProfile`：已经定义车辆宽度、长度、转弯半径、允许道路等级和安全缓冲字段。
- `StationRoadSnapService`、`StationCoordinateValidator`：支持站点靠近道路和坐标校验。
- `StationServicePosition`：已经有服务位、接入节点、服务方向和预约模型的基础。

问题不在于完全缺少类，而在于这些能力没有形成一条强制闭环，并且无效路线仍可能回到示意坐标运动。

## 3. 已确认的高风险逻辑

### 3.1 真实路线失败后仍可能执行示意路线

**相关文件：** `back/fsd-dispatch/src/main/java/com/fsd/dispatch/service/impl/ParkPilotSimulationServiceImpl.java`

当前流程大致是：

1. `routeToTarget()` 先调用 `planRoute()` 生成园区 X/Y 路线。
2. 对地理配送车辆，再调用 `beginGeoRouteForTarget()` 生成 GCJ-02 真实路线。
3. 如果真实路线无效或点数不足，就把 `state.geoFollower` 置空。
4. 后续 `moveVehicleAlongRoute()` 会继续使用 `state.route` 的园区路线。

这会造成一个非常直接的风险：

> 地理路线校验失败并不等于车辆停止；车辆可能继续沿另一套示意路线移动，而示意路线不一定经过真实道路。

必须补齐的规则：

- [ ] 地理配送车辆只允许执行通过真实道路校验的地理路线。
- [ ] 地理路线失败时进入 `ROUTE_BLOCKED`、`ROUTE_UNREACHABLE` 或 `MANUAL_INTERVENTION`，不能自动切换为示意路线。
- [ ] `state.geoFollower == null` 时，地理配送车辆不得继续执行 `state.route`。
- [ ] 示意车辆和地理配送车辆的路线执行器必须明确隔离。
- [ ] 前端必须显示“真实路线失败原因”，不能只显示车辆仍在移动。

### 3.2 空路线可能被当成有效校验结果

**相关文件：** `back/fsd-dispatch/src/main/java/com/fsd/dispatch/geo/RoadRouteCollisionValidator.java`

当前 `validate()` 对空折线返回 `valid(0D)`。这会造成语义矛盾：

- 没有路线不代表路线安全。
- 空折线无法证明没有穿建筑。
- 空折线无法证明车辆能到达。
- 上游如果只看 `invalid=false`，可能错误放行。

必须明确：

- [ ] 空路线必须是无效或不可达，而不是有效路线。
- [ ] 路线点数不足、总长度为零、起终点缺失分别返回独立失败原因。
- [ ] `valid=true` 的前提必须包括至少两个有效点、至少一条有效道路边和完整的碰撞检查。

### 3.3 路线首尾仍可能是未校验直线

**相关文件：** `back/fsd-dispatch/src/main/java/com/fsd/dispatch/service/impl/ParkRoutePlannerServiceImpl.java`

`buildRouteFromNodePath()` 会把原始起点和终点直接放入路线首尾，中间再放道路节点。即使节点之间沿路，以下两段仍可能不是道路：

- 原始起点 → 最近道路节点。
- 最近道路节点 → 原始终点。

如果原始点是建筑中心、POI 中心或未配置服务位的坐标，车辆就可能从建筑内部直接连到道路。

必须补齐：

- [ ] 起点必须是车辆当前位置或已校验的服务位。
- [ ] 终点必须是服务位，不允许直接使用建筑中心/POI 中心。
- [ ] 起点和终点都必须吸附到道路几何或服务位接入边。
- [ ] 吸附距离超过阈值时返回 `STATION_NOT_CONNECTED`，不能继续规划。
- [ ] 起终点连接段也必须进入建筑、河道和车辆包络碰撞检查。

### 3.4 当前碰撞检查不是完整的车辆包络检查

`RoadRouteCollisionValidator` 当前主要检查折线段是否与建筑/河道区域相交，以及采样点距离道路是否超过固定阈值。它还需要补充：

- [ ] 检查车辆中心线是否在可行驶道路区域内，而不是只接近某个道路折线。
- [ ] 用车辆外接矩形或圆形安全包络检查建筑物和障碍物。
- [ ] 安全包络使用车辆宽度、长度、车身朝向和 `safetyBufferMeters`。
- [ ] 检查整条路线的转弯半径，而不是只检查直线段。
- [ ] 检查路线是否在建筑物内部开始或结束。
- [ ] 检查路线是否经过步行街、人行区、消防通道、门禁区和禁停区。
- [ ] 检查建筑物 Polygon 内部点，而不只是检查边界相交。
- [ ] 检查路线折线被稀疏采样时是否漏掉短距离穿越。

## 4. 必须补齐的参数

### 4.1 路线请求级参数

每次规划必须能够完整记录以下字段：

| 参数 | 必填 | 说明 |
| --- | --- | --- |
| `parkId` | 是 | 路网所属园区，不能依赖默认园区猜测 |
| `mapVersion` | 是 | 使用的地图/路网版本 |
| `coordSystem` | 是 | 当前项目为 `GCJ-02` |
| `originType` | 是 | `VEHICLE_POSITION` / `SERVICE_POSITION` / `ACCESS_NODE` |
| `originLng`、`originLat` | 是 | 地理起点；必须说明是否已吸附 |
| `destinationType` | 是 | `SERVICE_POSITION` / `CHARGING_POSITION` / `PARKING_POSITION` |
| `destinationLng`、`destinationLat` | 是 | 地理终点；不得使用建筑中心代替服务位 |
| `originServicePositionId` | 条件必填 | 起点是服务位时必填 |
| `destinationServicePositionId` | 条件必填 | 终点是站点服务位时必填 |
| `originAccessNodeCode` | 条件必填 | 起点接入道路节点 |
| `destinationAccessNodeCode` | 条件必填 | 终点接入道路节点 |
| `snapDistanceMeters` | 是 | 起终点吸附距离 |
| `routeMode` | 是 | `REAL_ROAD` / `SCHEMATIC_ONLY`，地理车禁止混用 |
| `routeSourcePolicy` | 是 | `AMAP_ONLY` / `LOCAL_GRAPH_ONLY` / `AMAP_THEN_LOCAL` |
| `allowStraightLine` | 是 | 生产配送必须固定为 `false` |
| `avoidBuilding` | 是 | 必须固定为 `true` |
| `avoidPedestrianZone` | 是 | 必须固定为 `true`，除非车辆有特殊权限 |
| `avoidRiver` | 是 | 必须固定为 `true` |

### 4.2 道路路段参数

每条道路边不能只存起点和终点，还需要：

- [ ] `segmentId`、`fromNodeCode`、`toNodeCode`。
- [ ] `geometry`：真实道路中心线折点，而不是两点直线。
- [ ] `direction`：`FORWARD`、`REVERSE`、`BIDIRECTIONAL`。
- [ ] `widthMeters`：可行驶净宽，不是道路红线宽度。
- [ ] `roadClass`：主干、次干、支路、服务路、人行路、消防通道等。
- [ ] `accessState`：可行驶、仅服务、禁行、步行、装卸、充电接入等。
- [ ] `speedLimitKmh`：实际车辆限速。
- [ ] `turnRestriction`：禁止左转、右转、掉头、直行等。
- [ ] `turnRadiusMeters`：转弯处允许的最小半径。
- [ ] `allowedVehicleTypes`：允许车型或车辆组。
- [ ] `gateCode`、`gateOpenFrom`、`gateOpenUntil`：门禁约束。
- [ ] `blockedFrom`、`blockedUntil`：临时封路时间窗。
- [ ] `noStop`、`loadingOnly`：停靠和装卸规则。
- [ ] `geometryVersion`：道路几何版本。

### 4.3 建筑与障碍物参数

建筑物不能只存在于前端底图或固定禁行矩形中，必须是路线服务可读取的空间数据：

- [ ] `buildingId`、`buildingType`、`polygon`。
- [ ] `height`、`indoor`、`vehicleAccess`。
- [ ] `obstacleType`：建筑、围墙、河道、绿化、隔离带、施工区、停车障碍物。
- [ ] `bufferMeters`：按车辆尺寸膨胀后的安全边界。
- [ ] `effectiveFrom`、`effectiveUntil`：临时障碍物时间范围。
- [ ] `mapVersion`：障碍物必须与道路使用同一地图版本。
- [ ] `isHardBlock`：硬禁行还是只增加风险成本。
- [ ] `source`：OSM、高德、现场测绘、园区台账或人工修正。

特别注意：当前项目的 `PilotForbiddenZones.BUILDING_BLOCKS` 是近似矩形，适合早期试点保护，但不能代替真实建筑 Polygon。矩形边界偏大或偏小都会造成误判：偏小会穿建筑，偏大会误报无路。

### 4.4 站点与服务位参数

站点坐标不是车辆终点。每个可配送站点至少需要：

- [ ] `stationId`、`stationCode`、`stationType`。
- [ ] `servicePositionId`、`positionCode`。
- [ ] `positionLng`、`positionLat`，且坐标系为 `GCJ-02`。
- [ ] `accessNodeCode` 或 `accessEdgeId`。
- [ ] `serviceDirection`：车辆进入方向。
- [ ] `stopHeading`：车辆停靠朝向。
- [ ] `allowedVehicleTypes`。
- [ ] `vehicleWidthLimit`、`vehicleLengthLimit`。
- [ ] `capacityLimit`、`queueCapacity`。
- [ ] `avgServiceSeconds`、`serviceHours`。
- [ ] `gateCode`、`gateOpenFrom`、`gateOpenUntil`。
- [ ] `snapDistanceMeters`、`isInsideBuilding`、`isOnRoad`。
- [ ] `unreachableReason`、`unreachableUntil`。
- [ ] `positionStatus`：可用、占用、维护、封闭、待核验。

以下点不能直接作为车辆终点：

- [ ] 建筑物中心点。
- [ ] 高德综合 POI 中心点。
- [ ] 步行街中心点。
- [ ] 只有 X/Y 没有道路接入节点的站点。
- [ ] 只有经纬度没有服务方向的充电站。

### 4.5 车辆路由参数

每次路线规划都必须传入或明确读取车辆配置：

- [ ] `vehicleId`、`vehicleType`、`linkMode`。
- [ ] `widthCm`、`lengthCm`、`heightCm`、`weightKg`。
- [ ] `turningRadiusM`。
- [ ] `safetyBufferMeters`。
- [ ] `allowedRoadClasses`、`allowedVehicleTypes`。
- [ ] `maxSpeedKmh`、`currentSpeedKmh`。
- [ ] `batteryLevel`、`reserveBatteryLevel`。
- [ ] `chargingCapability`、`chargingTarget`。
- [ ] `currentHeading`、`currentLane` 或至少当前道路边。
- [ ] `currentMapVersion`。
- [ ] `manualOverride`、`emergencyMode`。

当前 `VehicleRoutingProfile` 已定义部分字段，但必须确认它真正参与每一次 `ParkRoadGraph` 建图和路线规划；“类里有字段”不等于“派车时已使用”。

## 5. 路线结果必须补齐的参数

规划服务返回的结果至少要包含：

| 参数 | 作用 |
| --- | --- |
| `routeId` | 路线唯一标识 |
| `mapVersion` | 路线使用的地图版本 |
| `source` | 高德、本地 OSM、园区图或人工路线 |
| `routeMode` | 真实道路或示意路线 |
| `nodePath` | 道路节点顺序 |
| `segmentPath` | 道路路段顺序 |
| `polyline` | 真实 GCJ-02 道路折线 |
| `distanceMeters` | 路线长度 |
| `estimatedTravelSeconds` | 按限速/速度计算的行驶时间 |
| `waitingSeconds` | 门禁、拥堵、服务位等待 |
| `turnCount` | 转弯数量 |
| `riskPoints` | 窄路、门禁、人车混行、消防通道等风险点 |
| `originSnapDistanceMeters` | 起点吸附距离 |
| `destinationSnapDistanceMeters` | 终点吸附距离 |
| `nearestRoadDistanceMeters` | 路线离道路最大距离 |
| `crossesBuilding` | 是否穿建筑 |
| `crossesRestrictedZone` | 是否穿禁行区 |
| `vehicleFit` | 车辆是否满足宽度/转弯/车型约束 |
| `collisionChecked` | 是否完成车辆包络校验 |
| `reservationStatus` | 多车资源是否锁定 |
| `invalid` | 路线是否不可执行 |
| `unreachableReason` | 无路、服务位缺失、道路封闭、车型不适配等具体原因 |

路线只返回两个点或只有一串坐标，不足以支撑真实车辆执行和审计。

## 6. 车辆执行层必须补齐的参数

车辆沿真实路线走，不只是拿到一串折线，还需要执行状态：

- [ ] `routeId` 和车辆当前执行路线绑定。
- [ ] `routeIndex` 或 `segmentIndex`。
- [ ] `traveledMeters` 和 `remainingMeters`。
- [ ] `currentLng`、`currentLat`、`currentHeading`。
- [ ] `lastMatchedSegmentId`：当前地图匹配到的道路边。
- [ ] `speedMps`、`positionTimestamp`、`eventSeq`。
- [ ] `offRouteDistanceMeters`。
- [ ] `rerouteCount`、`lastRerouteAt`、`rerouteReason`。
- [ ] `routeExecutionStatus`：执行、等待、偏航、阻塞、重规划、人工接管、完成。
- [ ] `telemetryCoordSystem`：车辆上报坐标系必须明确为 GCJ-02 或先转换。
- [ ] `mapVersion`：车辆执行版本与路线版本必须一致。

如果车辆上报只有位置，没有当前道路边、路线 ID、时间戳和偏航距离，系统无法判断它是沿路行驶还是穿越建筑。

## 7. 必须禁止的兜底行为

以下行为在真实无人车配送模式中必须禁止：

- [ ] 真实路线失败后自动切换为示意 X/Y 路线。
- [ ] 本地路网无路径时生成起点到终点的直线。
- [ ] 高德路线失败时只要有两个点就当成可执行路线。
- [ ] 站点没有服务位时使用站点中心坐标代替。
- [ ] 起点/终点吸附失败时继续使用原始坐标。
- [ ] 路线碰撞校验失败时只记录日志但继续移动。
- [ ] 路线版本与当前地图版本不一致时继续执行。
- [ ] MAPF 资源预约失败时把未预约路线当成已确认路线下发。
- [ ] 地理配送车辆在 `geoFollower` 为空时回退到 `state.route`。
- [ ] 空路线、点数不足或长度为零时返回“有效”。

允许的唯一降级方式是：

> 真实路线不可执行 → 车辆停止/安全等待 → 返回明确失败原因 → 人工处理或重新规划。

## 8. 推荐的完整路线流程

### 8.1 规划前

- [ ] 校验园区、地图版本、坐标系和路线模式。
- [ ] 校验车辆当前位置是否有效，是否属于当前地图范围。
- [ ] 校验站点、服务位和接入节点是否有效。
- [ ] 校验车辆宽度、长度、转弯半径和电量。
- [ ] 校验道路、建筑和禁行区数据版本一致。

### 8.2 路线计算

- [ ] 起点吸附到当前可行驶道路边。
- [ ] 终点吸附到服务位接入边。
- [ ] 根据车辆 profile 过滤道路。
- [ ] 根据方向、门禁、时间窗和临时封路过滤道路。
- [ ] 使用 Dijkstra/A* 或高德真实驾车路线生成节点/路段路径。
- [ ] 计算真实道路折线，不使用节点之间的屏幕直线替代真实几何。
- [ ] 计算路线长度、限速 ETA、转弯风险和等待成本。

### 8.3 安全校验

- [ ] 检查路线中心线不进入建筑/河道/禁行区。
- [ ] 对建筑和障碍物按车辆包络做膨胀。
- [ ] 检查车辆包络与膨胀障碍物不相交。
- [ ] 检查路线每个采样点都落在可行驶道路区域。
- [ ] 检查起点和终点不在建筑内部。
- [ ] 检查服务方向和转弯半径。
- [ ] 检查路线来源不是 `STRAIGHT_LINE`。
- [ ] 校验失败立即拒绝执行。

### 8.4 执行与监控

- [ ] 将 `routeId`、`mapVersion`、`segmentPath` 下发给车辆或仿真器。
- [ ] 按道路折线弧长推进，不按两个屏幕点直接插值。
- [ ] 实时地图匹配车辆位置到道路边。
- [ ] 超过偏航阈值后暂停执行并重规划。
- [ ] 重规划仍使用同一车辆 profile 和障碍物版本。
- [ ] 车辆到达服务位后才进入装卸/充电状态。
- [ ] 记录完整轨迹和路线审计信息。

## 9. 参数缺失优先级

### P0：必须补齐，否则不能保证不穿建筑

- [ ] 地理路线失败不能回退示意路线。
- [ ] 空路线不能标记有效。
- [ ] 起点和终点服务位/道路接入节点。
- [ ] 建筑物真实 Polygon 或可信障碍物数据。
- [ ] 路线起终点吸附距离和吸附结果。
- [ ] `allowStraightLine=false`。
- [ ] `routeMode=REAL_ROAD`。
- [ ] `collisionChecked=true`。
- [ ] `mapVersion` 全链路一致。

### P1：必须补齐，否则路线可能不适合车辆

- [ ] 道路几何、方向、宽度、限速和临时封路。

## 10. 最小验收用例

- [ ] 起点位于道路上、终点为取货服务位：路线沿道路到服务位。
- [ ] 终点坐标放在建筑中心：系统拒绝，不自动穿建筑。
- [ ] 起点/终点距道路超过阈值：系统返回未接入，不生成路线。
- [ ] 真实路线碰撞校验失败：车辆停止，不执行示意路线。
- [ ] 主路封闭：车辆沿备用路段绕行。
- [ ] 道路宽度小于车身包络：系统返回车型不适配。
- [ ] 门禁未开放：车辆在门前等待，不穿越禁区。
- [ ] MAPF 预约失败：显示未锁定路线，不直接下发执行。
- [ ] 车辆偏离路线：超过阈值后暂停并重规划。
- [ ] 地图版本切换：旧路线不能静默套用新障碍物数据。
- [ ] 充电点：车辆进入充电服务位，而不是进入公共 POI 中心。

## 11. 最终回答：你目前最可能少传入的参数

结合当前项目，最需要优先确认并补齐的是：

1. **服务位 ID 和道路接入节点**：目前不能只传站点经纬度。
2. **路线模式**：必须明确地理车使用 `REAL_ROAD`，不能和示意车共用默认路线。
3. **路线失败策略**：真实路线失败必须停止，不能回退 `state.route`。
4. **建筑/障碍物 Polygon 版本**：不能只依赖静态近似矩形。
5. **车辆外形参数**：宽、长、转弯半径、安全缓冲必须参与校验。
6. **路段几何和方向**：不能只用道路节点两点连线。
7. **地图版本和路线来源**：防止路线和障碍物来自不同版本。
8. **车辆当前道路边、偏航距离和重规划状态**：否则无法知道车辆是否真正沿路。
9. **空路线/无效路线状态**：必须与“安全可执行”彻底区分。
10. **服务位停靠朝向、门禁和时间窗**：否则即使路线不穿建筑，也可能无法实际到站。

## 12. 实施前的决策原则

在补齐上述参数前，不建议继续通过调整前端路线颜色、增加地图标签或修改车辆动画来掩盖问题。正确顺序应是：

> **服务位定点 → 道路接入 → 车辆 profile → 障碍物版本 → 路线生成 → 碰撞校验 → 路线锁定 → 车辆执行 → 偏航重规划。**

只要其中任何一步失败，系统就应该让车辆等待并给出原因，而不是让车辆沿另一套坐标路线继续移动。
