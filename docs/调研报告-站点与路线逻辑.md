## 2. 前端站点（路线图 3.1 – 3.3）

### 3.1 真实/示意站点使用两套分类规则 —— ✅ 已确认

**证据 `front/src/maps/stationLayers.ts`：**

```ts
// L4-14 真实/示意判定完全依赖编码与 area
export const GEO_DELIVERY_AREA = 'ZJF'
export function isGeoDeliveryStation(s): boolean {
  if (station.area === GEO_DELIVERY_AREA) return true
  return (station.stationCode ?? '').startsWith('ZJF-')
}
export function isSchematicParkStation(s): boolean { return !isGeoDeliveryStation(s) }
```

```ts
// L96-105 仅调度/回充站点由硬编码编码决定
export function isZjfDispatchOnlyStation(s): boolean {
  const code = s.stationCode ?? ''
  return code === 'ZJF-IDLE-01' || code.startsWith('ZJF-CHG-')
}
```

- 移动端下单：`filterMobileOrderStations`（L103）`= filterGeoDeliveryStations` 再排除 CHG/IDLE。
- 工作台态势：`filterWorkbenchSituationStations`（L126）默认 8 运营站，可选 IDLE/CHG。
- 示意地图：`filterSchematicStations`（L42）。
- 角色着色：`workbenchStationRole`（L149）按 `ZJF-PICK-/DROP-/EXPRESS-/CHG-` 前缀分流。

**风险已全部坐实：**
1. 是否"真实"取决于 `area==='ZJF'` 或 `ZJF-` 前缀，而非数据库中的地图版本/用途/几何。
2. 新增站点编码不符合约定会被误显或过滤（如 `syncDefaultOrderStations` 硬编码 `'ZJF-PICK-'`、`'ZJF-DROP-01'`、`'A1'`、`'B1'` 兜底，L265-281）。
3. 类型数据模型 `ParkStation` 仅有 `deliveryZone`（GEO/SCHEMATIC/GENERAL）与 `status`，**无 `orderable`/`dispatchable` 显式能力字段**，能力完全由编码推导。
4. `filterStationsByDeliveryZone`（L83）实现了基于 `deliveryZone` 字段的另一套规则，但全局搜索显示**该函数未被任何 `.vue`/`.ts` 调用**（仅 `maps/index.ts` 导出）。这是"同一站点多套过滤规则"的真实隐患：一旦死代码被启用，`GENERAL` 站点会双显。

- **无单一权威坐标**：geo 地图优先 `coordLng/coordLat`，回退 `parkXYToGcj02(x,y)`；示意地图 (`ParkMiniMap.vue`) 直接用 `x/y`。两套坐标各行其是。
- **响应模型已预留 `anchorNodeCode` / `serviceDirection` / `unreachableReason`**（`@/types/infrastructure.d.ts`），但 `handleSave` 的 payload（L371-385）**不包含这三项**，表单没有编辑入口，服务位配置流程断裂——路线图中"响应模型存在但表单未形成完整配置"完全属实。

### 3.3 地图应显示服务位而非只显示站点 —— 🟡 部分确认

全仓 Grep `servicePosition|accessNode|chargePoint`：**零匹配 `servicePosition`**。前端根本没有"服务位"概念。

- "接入道路节点"仅以站点字段 `anchorNodeCode` 静态列展示。
- "充电桩"对应 `AdminChargingPile`，有类型与报表表格，但**地图图层（geo/schematic）不渲染为点**，不与站点联动。
- 不存在"站点 → 服务位 → 接入道路 → 当前可达性"的钻取链路；`anchorNodeCode/serviceDirection/unreachableReason` 只是只读列。

结论：路线图 3.3 的设计诉求当前**完全未实现**，不是"显示错了"，而是"概念未建模"。

## 3. 后端逻辑（路线图 4.1 – 4.10）

> 后端根：`back/fsd-dispatch/src/main/java/com/fsd/dispatch/`

### 4.1 起终点未吸附到道路 —— ✅ 已确认（P0，已逐行复核）

**证据 `service/impl/ParkRoutePlannerServiceImpl.java`：**

```java
// L64-80 buildRouteFromNodePath
route.add(point("START", startX, startY, null, null));   // L71 原始坐标，lng/lat=null
for (String code : nodePath) {
    ParkRoadGraph.NodeView node = graph.node(code);
    if (node != null) {
        route.add(point(code, node.x(), node.y(), node.coordLng(), node.coordLat())); // L75 中间节点带 GPS
    }
}
route.add(point("END", endX, endY, null, null));          // L78 原始坐标，lng/lat=null
```

START/END 直接采用调用方原始 `startX/startY/endX/endY` 且 `lng/lat=null`，仅中间节点带道路 GPS。注释自称"供下游 pathLength 用 haversine 算真实长度"，但**首尾直线段本身未做任何道路/障碍物校验**——即"最后一段穿建筑/停到路中心"的风险在代码层面确实存在。

### 4.2 空路网 `isReachable()` 返回 true —— ✅ 已确认（P0，已逐行复核）

```java
// L51-55 buildRoute
ParkRoadGraph graph = resolveGraph(parkId);
if (graph.isEmpty()) {
    throw new BusinessException("PARK_ROAD_NETWORK_EMPTY", "园区路网数据为空...");
}

// L100-103 isReachable
ParkRoadGraph graph = resolveGraph(parkId);
if (graph.isEmpty()) {
    return true;   // ← 空路网却报"可达"
}
```

**逻辑矛盾坐实**：在 `DispatchVehicleAssignServiceImpl.estimateRouteDistance` 中先 `isReachable()→true` 再 `buildRoute()→抛异常`，异常未被捕获向上冒泡。前端若先调可达性接口放行走，派单时才失败——正是路线图担心的"提前放行"场景。空路网应视为不可达/未配置。

### 4.3 MAPF 兜底语义不统一 —— 🟡 部分确认

`mapf/MapfRoutePlannerService.java` `planAndReserve()`（L74-80）：多车预约失败后调 `buildRoute()` 兜底并返回 `reserved=false`（L77）。

- `reserved=false` 实际只代表"冲突未预约"，但**"找不到路"（无路径）与"找到路但无时空资源"（冲突）未区分**：兜底路径内 `buildRoute→shortestNodePath` 无路时抛 `PARK_ROUTE_NOT_FOUND`（未捕获，向上抛），`reserved=false` 因此无法表达"无路"。
- 消费方 `DispatchVehicleAssignServiceImpl.selectWithMapfReservation`（L215）仅判 `plan.isSuccess() && plan.isReserved()`；`MapfRoutePlanResult` 无 `reason` 字段，无法区分两类失败。
- 路线图"应区分找不到路 vs 找到路但无法获得时空资源"的诉求成立。

### 4.4 示意路网回退仍有 STRAIGHT_LINE —— 🟡 部分确认（比担心的轻）

`geo/local/LocalPilotRoadGraphService.java` `planDrivingRoute` 确实在多处返回 `STRAIGHT_LINE` 空结果（L246 空起终点 / L267 无走廊 / L272 走廊<4 点）；`geo/ChainedRoadRouteService.java`（L64）amap+local 均失败时 `new RoadRouteResult(List.of(), 0D, STRAIGHT_LINE)`，并保留 `straightLine()` 构造函数。

**关键缓解**：生产派单入口 `DispatchVehicleAssignServiceImpl.selectBestVehicle` 直接调 `parkRoutePlannerService.buildRoute/isReachable`，**不经过 `ChainedRoadRouteService`**；STRAIGHT_LINE 兜底只存在于仿真/导航路径（`ParkPilotSimulationServiceImpl`）。即"直线进入真实车辆执行队列"的核心风险在当前调用链上**不成立**——但仿真链路仍可能产生直线，需确认仿真结果不外溢到执行队列。响应 `RoadRouteResult` 带 `source`/`invalid`/碰撞字段，但**无"不可达原因"字段**。

### 4.5 多套路网事实源 —— ✅ 已确认（P0）

```java
// L125-140 resolveGraph
List<RoadNodeEntity> dbNodes = roadNodeMapper.selectList(... eq("status","ACTIVE") ...);
if (!dbNodes.isEmpty()) {
    return ParkRoadGraph.fromDatabase(dbNodes, dbSegments);  // DB 优先
}
return ParkRoadGraph.fromYaml(parkPilotProperties);          // YAML 兜底
```

- 派单路径走 `ParkRoutePlannerServiceImpl`（DB / YAML）。
- `LocalPilotRoadGraphService` 仍保留手工走廊 `segmentCache`（`initSegments`）与固定锚点 `stationAnchors`（`initAnchors`）。
- OSM 子图来自 `OsmPilotGeoRepository`。
- **四套事实源并存**：①DB ②YAML ③OSM ④手工走廊；其中 ①②④在仿真路径、①③在派单路径，两套链路事实源不一致，更新其一不更新另一会导致前后端路线不一致。
- **地图版本**：全后端无 `MapVersion`/`map_version`/`activeVersion` 引用，`resolveGraph` 未绑定激活版本。路线图中"地图版本表需验证是否强制绑定激活版本"→ 当前**未绑定**。

### 4.6 坐标距离计算模式混用 —— ✅ 已确认（P1）

`road/ParkRoadGraph.java` `NodeView.distanceTo()`（L236-241）：两端均有 GPS 用 `haversineMeters`，否则退回欧氏 `distanceTo(x,y)`。

结合 4.1：START/END `lng/lat=null`，中间节点带 GPS。`DispatchVehicleAssignServiceImpl.pathLength`（L439-450）逐段判断"两端都有 GPS 才 haversine，否则 `hypot`（示意）"——START→首节点段若 GPS 未补全，即**米与像素混算**。`enrichRouteEndpointsGeo`（L276-278）解析失败静默回退欧氏。距离/ETA 不可信风险成立。

### 4.7 路段状态有过滤，车辆几何约束未完成 —— 🟡 部分确认

- 路段状态确有：`fromDatabase` 处理 `status`、BLOCKED/PEDESTRIAN_ONLY、临时封路 `isCurrentlyBlocked`、direction 建边（L85-132）。
- **车辆几何约束缺失**：`fromDatabase` 注释声称"vehicle-specific filtering via `buildForVehicle`"，但全代码 `buildForVehicle` **仅在注释出现，方法不存在**；无对路段 `width/height/allowedType` 参与选路的代码。`matchesRequiredVehicleType` 仅比对 `DispatchRouteEntity.requiredVehicleType`，非路段几何约束。
- 结论：限高/限宽/转弯半径/车辆类型对寻路**无实际约束**，路线图担心成立。

### 4.8 服务位预留并发一致性 —— ❌ 未确认 / 不成立（比担心的更糟）

`service/StationServicePositionLockService.java` 提供 `acquire/lock/reserve/release/sweep`（L48-183），但全局搜索显示**除本类外无任何调用方**（仅 `target` 产物命中）。

- 服务位互斥在当前派单链路**实际未生效**。
- 无 `@Version`/DB unique 证据（仅 select-then-insert，存在并发双写窗口）。
- `reservedUntil` 自动释放依赖 `sweepExpiredReservations`，但无调度器调用 → 永不触发。
- `cancelTask`（L470-520）仅 `releaseVehicle` + `parkingFacilityService.releaseByVehicle`，**不触碰服务位预约**；断线/异常/充电超时同理。

路线图假设"已有实体与管理入口，需验证并发"，实际是"入口未接线"，风险等级应上调到 P0 级"资源锁定形同虚设"。

### 4.9 派车状态机/幂等/暂停受控 —— 🟡 部分确认

- 状态机 `service/impl/DispatchTaskStateServiceImpl.java` `ALLOWED_TRANSITIONS` + `assertCan*`（L30-117）受控，非法迁移抛 `DISPATCH_TASK_STATUS_INVALID`。
- 幂等：`autoAssignTask`（L233）`acquireTaskLock` 防并发重入；同订单 `assertCanCreateTask` 防重。
- **漏洞**：
  1. `manualAssignTask`（L384-462）仅 `assertCanManualAssign`（状态），直接 `occupyVehicle` 置 ASSIGNED——**绕过电量/车型/路网/服务位校验**。
  2. **暂停派单只覆盖自动**：`autoAssignTask`（L272）、`selectBestVehicle`（L87）检查 `isDispatchPaused`，但 `manualAssignTask` **未检查**——暂停期间手动派车仍可进行。
  3. 紧急插队无独立入口（手动派车即事实插队）。

### 4.10 异常文案乱码 —— ❌ 未确认（无乱码，但异常码未集中枚举）

未检出中文乱码（注释 UTF-8、消息英文常量）。`BusinessException`（code+message）code 为自由字符串（如 `PARK_ROAD_NETWORK_EMPTY`），风格统一但**未集中枚举**；任务失败原因有 `DispatchAssignFailReason` 枚举，路网/路线类 code 仍散落字符串。路线图"统一异常码与中文展示文案"诉求部分成立——需补中文展示层与不可达原因字段（呼应 4.3/4.4）。

### 5.4 STRAIGHT_LINE 禁令现状
