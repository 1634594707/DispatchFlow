## 1. 园区地图配置手册

### 1.1 园区元数据字段

园区元数据存储在 `t_park` 表，每个园区一行。阶段七 7.3 将硬编码的 `ZJF_PILOT_GEO` 迁入数据库，支持多园区配置。

| 字段 | 类型 | 说明 | 示例 |
|---|---|---|---|
| `id` | BIGINT | 主键 | `1` |
| `park_code` | VARCHAR | 园区编码（唯一） | `ZJF_DIESHIQIAO` |
| `park_name` | VARCHAR | 园区名称 | `找家纺网 · 叠石桥短驳试点` |
| `default_flag` | TINYINT | 是否默认园区（1=是） | `1` |
| `center_lng` | DECIMAL | 园区中心经度（GCJ-02） | `121.080354` |
| `center_lat` | DECIMAL | 园区中心纬度（GCJ-02） | `31.961977` |
| `anchor_lng` | DECIMAL | **锚点经度**（schematic 坐标系原点，GCJ-02） | `121.080354` |
| `anchor_lat` | DECIMAL | **锚点纬度**（schematic 坐标系原点，GCJ-02） | `31.961977` |
| `park_width_meters` | DECIMAL | 园区宽度（米） | `1570` |
| `park_height_meters` | DECIMAL | 园区高度（米） | `470` |
| `map_width` | INT | schematic 地图宽度（像素） | `1200` |
| `map_height` | INT | schematic 地图高度（像素） | `800` |
| `map_provider` | VARCHAR | 地图提供商 | `AMAP` |
| `scenario_code` | VARCHAR | 场景标识 | `ZJF_DIESHIQIAO_PILOT` |
| `deleted` | TINYINT | 软删除（0=有效） | `0` |

### 1.3 地图层级 L0/L1/L2

阶段七 7.5 实现了三级地图层级切换，前端 `AmapGeoMap.vue` 右上角提供切换器：

| 层级 | 名称 | 缩放级别 | 显示元素 | 用途 |
|---|---|---|---|---|
| L0 | 产业带 | ~10 | 20km 覆盖圈 | 区域概览，不显示车辆/站点细节 |
| L1 | 试点 | ~14 | 园区围栏 + 车辆 + 轨迹 | 默认层级，日常运营监控 |
| L2 | 站点 | ~16 | 站点 marker + 详细路径 | 站点级细节查看 |

**前端集成**：父组件通过 `v-model:map-level` 双向绑定当前层级，可根据层级自行过滤传入的 markers/polygons。

### 1.4 离线地图缓存

阶段七 7.6 通过 Service Worker 实现高德瓦片离线缓存（`front/src/sw.ts`）：
- 缓存域名：`*.amap.com`、`*.autonavi.com`、`webapi.amap.com`、`restapi.amap.com`
- 缓存策略：StaleWhileRevalidate（优先缓存，后台更新）
- 缓存上限：500 张瓦片
- 过期时间：30 天

**验证弱网可用性**：Chrome DevTools → Network → Offline → 刷新地图页面，应仍能显示已缓存的瓦片。

### 1.5 高德 Key 配置

**构建时配置**（`front/.env.local`）：
```bash
VITE_AMAP_KEY=your_amap_key_here
VITE_AMAP_SECURITY_CODE=your_security_code_here
VITE_MAP_PROVIDER=AMAP
VITE_AMAP_DEFAULT_CENTER=121.080354,31.961977
VITE_AMAP_DEFAULT_ZOOM=15
```

**运行时配置**（`front/public/runtime-config.js`，部署后可修改无需重新构建）：
```javascript
window.__DISPATCHFLOW_RUNTIME_CONFIG__ = {
  VITE_AMAP_KEY: 'your_amap_key_here',
  VITE_AMAP_SECURITY_CODE: 'your_security_code_here',
  VITE_MAP_PROVIDER: 'AMAP',
}
```

**高德控制台白名单**：必须将部署域名加入高德控制台的安全域名白名单，否则地图加载会报 `INVALID_USER_SCODE` 或 `USERKEY_PLAT_NOMATCH` 错误。

## 2. 围栏设计规范

### 2.1 围栏类型与响应级别

阶段六 6.1/6.2 为 `t_park_geofence` 表增加 `response_level` 和 `buffer_meters` 字段：

| 字段 | 类型 | 可选值 | 默认值 | 说明 |
|---|---|---|---|---|
| `fence_type` | VARCHAR | `BOUNDARY` / `RESTRICTED` | — | 围栏类型：边界/禁区 |
| `response_level` | VARCHAR | `INFO` / `WARN` / `BLOCK` | `WARN` | 响应级别 |
| `buffer_meters` | DECIMAL(10,2) | 0-1000 | `15.00` | GPS 缓冲距离（米） |

**响应级别行为**：

| 级别 | 行为 | 适用场景 |
|---|---|---|
| `INFO` | 仅记录日志，不写异常、不触发自动化规则 | 信息采集，如车辆经过某区域 |
| `WARN` | 记录异常 + 触发自动化规则（默认行为） | 标准边界围栏，车辆越界告警 |
| `BLOCK` | 记录异常 + 触发自动化规则 + `markUnavailable` 紧急停车 | 禁区围栏，车辆进入危险区域 |

**`fence_type` 与 `response_level` 的默认映射**（V40 迁移）：
- `RESTRICTED` → `BLOCK`
- `BOUNDARY` → `WARN`
- 其他 → `WARN`

### 2.2 GPS 缓冲距离设计

`buffer_meters` 用于抑制 GPS 漂移导致的误报。车辆位于围栏外但距边界小于 `buffer_meters` 时，不触发 GEOFENCE_EXIT。

| 场景 | 推荐 buffer_meters | 理由 |
|---|---|---|
| 室外开放区域 | 15（默认） | 标准 GPS 精度 5-10m，15m 留余量 |
| 高层建筑密集区 | 25-30 | 多径效应增强，GPS 漂移更大 |
| 室内/地下 | 50+ | GPS 信号弱，漂移可能超过 30m |
| 精密作业区 | 5-10 | 有 RTK 或差分 GPS，精度高 |

**重要约束**：
- `buffer_meters` 设为 0 表示无缓冲（任何越界立即告警）— 仅推荐在 RTK 场景使用
- `buffer_meters` 过大（>50）会导致真正的越界被误抑制

### 2.3 围栏多边形规范

围栏多边形存储在 `t_park_geofence.polygon_json` 字段，格式为 GCJ-02 坐标的 JSON 数组：

```json
[[lng1,lat1],[lng2,lat2],[lng3,lat3],...]
```

**设计要求**：
- 最少 3 个顶点（三角形）
- 顶点顺序：**逆时针**（GeoJSON 规范，外环）
- 坐标精度：6 位小数（约 0.1 米）
- 多边形必须**简单**（不自交）
- 单个围栏顶点数建议 ≤ 32（性能考虑）

**叠石桥试点 5 分区围栏**（V37 迁移）：

| 围栏编码 | 围栏名称 | 颜色 | 用途 |
|---|---|---|---|
| `ZJF-ZONE-CORE-SOUTH` | 家纺城核心南排区 | `#1677ff` | 门市取货 |
| `ZJF-ZONE-CORE-NORTH` | 家纺城核心北排区 | `#52c41a` | 仓库 |
| `ZJF-ZONE-HUB` | 代发仓集散区 | `#fa8c16` | 代发仓主枢纽 |
| `ZJF-ZONE-EAST` | 东排代拿仓区 | `#722ed1` | 代拿仓 |
| `ZJF-ZONE-EXPRESS` | 快递接驳物流区 | `#eb2f96` | 快递网点接驳 |

### 2.4 围栏自检调度

阶段六 6.3 实现了月度自检调度器 `GeofenceSelfCheckScheduler`：
- Cron：`0 0 3 1 * ?`（每月 1 日 03:00）
- 检查项：
  1. 所有 `ACTIVE` 围栏是否包含至少 1 个站点
  2. 围栏多边形是否有效（不自交、≥3 顶点）
- 检查结果写入日志，空围栏和无效多边形会告警

**配置**（`application.yml`）：
```yaml
fsd:
  automation:
    geofence-self-check-cron: "0 0 3 1 * ?"  # 默认每月 1 日 03:00
```

## 3. 坐标校准流程

### 3.1 坐标系说明

| 坐标系 | 用途 | 说明 |
|---|---|---|
| **WGS-84** | GPS 原始坐标 | 国际标准，Google Maps/OSM 使用 |
| **GCJ-02** | 中国国内地图坐标 | 高德/腾讯/谷歌中国使用，对 WGS-84 加密偏移 |
| **BD-09** | 百度地图坐标 | 在 GCJ-02 基础上再次偏移 |
| **schematic (park x/y)** | 园区示意图坐标 | 像素坐标，原点在地图左上角 |

**DispatchFlow 全程使用 GCJ-02**（与高德地图一致）。若从 GPS 设备获取的是 WGS-84，需先转换为 GCJ-02。

### 3.2 锚点校准流程

锚点（anchor）是 schematic 坐标系与 GCJ-02 坐标系的对应关系原点。校准步骤：

**步骤 1：采集控制点**

至少采集 3 个已知站点的真实 GCJ-02 坐标（用高德地图 App 在现场定位）：
- 站点 A：`(lng_a, lat_a)` + 对应的 schematic `(x_a, y_a)`
- 站点 B：`(lng_b, lat_b)` + 对应的 schematic `(x_b, y_b)`
- 站点 C：`(lng_c, lat_c)` + 对应的 schematic `(x_c, y_c)`

**步骤 2：反算锚点**

使用最小二乘法反算最优锚点 `(anchor_lng, anchor_lat)`，使三个站点的转换误差最小。叠石桥试点使用 3 个站点反算结果：
```
ZJF-PICK-01 (121.074453, 31.960396)
ZJF-DROP-01 (121.079762, 31.963627)
ZJF-DROP-02 (121.087005, 31.961780)
→ 反算结果：anchor_lng=121.080354, anchor_lat=31.961977
```

**步骤 3：写入数据库**

```sql
UPDATE t_park
SET anchor_lng = 121.080354,
    anchor_lat = 31.961977,
    park_width_meters = 1570,
    park_height_meters = 470
WHERE park_code = 'ZJF_DIESHIQIAO';
```

**步骤 4：验证转换精度**

调用坐标转换 API 验证：
```bash
# 前端 schematic → GCJ-02
curl "http://your-server:501/api/admin/park/geo/transform?parkX=600&parkY=400" \
  -H "Authorization: Bearer your_admin_token"

# 反向：GCJ-02 → schematic
curl "http://your-server:501/api/admin/park/geo/transform?longitude=121.080354&latitude=31.961977" \
  -H "Authorization: Bearer your_admin_token"
```

**验收标准**：所有已知站点往返转换偏差 < 20 米（阶段九 9.1 回归测试验证）。

### 3.3 坐标转换 API

| 接口 | 方法 | 说明 |
|---|---|---|
| `/api/admin/park/geo/transform` | GET | 双向转换（parkX/parkY ↔ longitude/latitude） |
| `/api/admin/park/calibration-points` | GET | 列出所有校准点（站点坐标对） |
| `/api/admin/park/metadata` | GET | 获取园区元数据（锚点、尺寸） |

### 3.4 WGS-84 → GCJ-02 转换

若车辆设备上报的是 WGS-84 坐标，需在后端 `VehicleReportServiceImpl` 中先转换为 GCJ-02 再存储。转换算法参考 `com.fsd.dispatch.geo.CoordinateTransformService`。

## 4. 新增园区完整流程

### 4.1 准备工作

- [ ] 实地勘察园区，采集至少 3 个控制点的 GCJ-02 坐标
- [ ] 测量园区尺寸（宽度、高度，单位：米）
- [ ] 绘制园区 schematic 地图（PNG/SVG，记录像素尺寸）
- [ ] 规划围栏分区（每个分区一个多边形）
- [ ] 规划站点位置（取货点、送货点、充电站、待命区）

### 4.2 数据库配置

**步骤 1：插入园区记录**
```sql
INSERT INTO t_park (park_code, park_name, default_flag, center_lng, center_lat,
                    anchor_lng, anchor_lat, park_width_meters, park_height_meters,
                    map_width, map_height, map_provider, scenario_code, deleted)
VALUES ('NEW_PARK_CODE', '新园区名称', 0,
        121.080354, 31.961977,  -- 替换为实际锚点
        121.080354, 31.961977,
        1570, 470,              -- 替换为实际尺寸
        1200, 800,              -- 替换为 schematic 像素尺寸
        'AMAP', 'NEW_SCENARIO_CODE', 0);
```

**步骤 2：插入站点记录**
```sql
INSERT INTO t_station (park_id, station_code, station_name, area,
                       coord_lng, coord_lat, x, y, station_type, status, deleted)
VALUES (/* park_id */, 'NEW-PICK-01', '取货点1', 'NEW',
        121.074453, 31.960396,  /* GCJ-02 坐标 */
        600, 400,               /* schematic 坐标 */
        'PICKUP', 'ACTIVE', 0);
```

**步骤 3：插入围栏记录**
```sql
INSERT INTO t_park_geofence (park_id, fence_code, fence_name, fence_type,
                             response_level, buffer_meters, polygon_json, status, deleted)
VALUES (/* park_id */, 'NEW-ZONE-01', '分区1', 'BOUNDARY',
        'WARN', 15.00,
        '[[121.071812,31.960126],[121.073405,31.959762],...]',
        'ACTIVE', 0);
```

**步骤 4：插入路网节点和路段**
```sql
-- 路网节点
INSERT INTO t_road_node (park_id, node_code, lng, lat, x, y, deleted)
VALUES (/* park_id */, 'R1', 121.075, 31.961, 100, 120, 0);

-- 路网路段（支持单行道，阶段七 7.1）

VALUES (/* park_id */, 'R1', 'R2', 'BIDIRECTIONAL', NULL, 0);
```

`direction` 可选值：`BIDIRECTIONAL`（双向，默认）、`FORWARD`（仅正向）、`REVERSE`（仅反向）。

### 4.3 前端配置

**步骤 1：高德 Key 配置**

在 `front/.env.local` 或 `front/public/runtime-config.js` 中配置高德 Key（见 1.5 节）。

**步骤 2：地图层级配置**

前端默认从 `/api/admin/park/metadata` 获取园区元数据，无需额外配置。若 API 不可用，自动回退到 `ZJF_PILOT_GEO` 硬编码。

**步骤 3：验证**

访问前端 `/system/config-check` 页面，执行试点配置自检。所有检查项应通过。

- [ ] `/api/admin/park/metadata` 返回正确的园区元数据

## 5. 故障排查

### 5.1 地图加载失败

**症状**：前端显示"高德 Key 未配置"或地图空白

**排查**：
1. 检查 `front/.env.local` 或 `runtime-config.js` 中 `VITE_AMAP_KEY` 和 `VITE_AMAP_SECURITY_CODE` 是否配置
2. 检查高德控制台是否将当前域名加入白名单
3. 浏览器控制台查看具体错误码：
   - `INVALID_USER_SCODE`：安全密钥错误
   - `USERKEY_PLAT_NOMATCH`：Key 平台类型不匹配（需 Web 端 JS API）
   - `INVALID_USER_DOMAIN`：域名未白名单

### 5.2 站点位置偏差大

**症状**：站点 marker 与实际位置偏差 > 20 米

**排查**：
1. 调用 `/api/admin/park/metadata` 确认 `anchor_lng/lat` 和 `park_width/height_meters` 正确
2. 调用 `/api/admin/park/geo/transform?longitude=xxx&latitude=xxx` 验证转换结果
3. 重新执行锚点校准流程（第 3.2 节）
4. 运行 `MapAccuracyRegressionTest` 验证转换精度

### 5.3 围栏告警误报

**症状**：车辆在围栏内却触发 GEOFENCE_EXIT，或越界后不告警

**排查**：
1. 检查 `t_park_geofence.buffer_meters` 是否过大（建议 15m）
2. 检查 `polygon_json` 顶点顺序是否为逆时针
3. 检查车辆上报的坐标是否为 GCJ-02（非 WGS-84）
4. 运行 `GeofenceBreachServiceImplTest` 验证边界检测逻辑
5. 查看 `GeofenceSelfCheckScheduler` 的月度自检日志

### 5.4 路径规划不可达

**症状**：两个站点间无法生成路径

**排查**：
1. 检查 `t_road_node` 和 `t_road_segment` 是否完整（节点和路段是否连接）
2. 检查 `t_road_segment.direction` 是否正确（单行道方向）
3. 检查 `t_road_segment.connecting_park_id` 是否正确（跨园区路段）
4. 运行 `ParkRoutePlannerServiceImplTest` 验证路径规划逻辑

### 5.5 充电阈值不生效

**症状**：修改 Redis 中充电阈值后车辆行为未变化

**排查**：
1. 确认 Redis key 正确：`fsd:config:energy:return-to-charge-threshold`（见运维手册-评分权重与能量阈值.md）
2. 确认 Redis 值为 0-100 的整数百分比字符串
3. 等待 5 秒（本地缓存 TTL）后再次观察
4. 调用 `FleetEnergyThresholdResolver.refresh()` 强制刷新缓存
