# DispatchFlow 最终更新路线图

> 版本：2026-07-18  
> 状态：方案确认阶段，**本文件不代表已完成代码实现**。  
> 后续实施优先以本路线图和 `真实配送范围与地图视觉规范_2026-07-18.md` 为准。

## 1. 最终目标

DispatchFlow 需要从“能在地图上显示车辆和站点”升级为“能证明车辆沿可行驶道路执行任务”：

- 订单只能选择有真实服务位的配送目标。
- 路线必须来自当前地图版本的道路图或可信地图服务。
- 路线起点、终点和中间路段都通过道路、建筑和车辆包络校验。
- 真实路线失败时车辆停止等待，不回退到示意线或起终点直线。
- 前端能解释车辆为什么能走、为什么不能走、当前路线来自哪里。
- 主工作台只承担实时调度，不再承担“页面分工”宣传说明。

## 2. 最终事实基线

### 2.1 坐标基线

- [ ] 坐标存储和高德底图统一使用 `GCJ-02`。
- [ ] L1 局部投影/路网基点：`[121.080354, 31.961977]`。
- [ ] L0 主市场语义中心：`[121.080800, 31.964305]`。
- [ ] 家纺城三期参考点：`[121.083349, 31.963624]`。
- [ ] 叠石桥步行街参考点：`[121.078896, 31.963593]`。
- [ ] 纺都大道 188 号外部参考点：`[121.099689, 31.954391]`。
- [ ] `ZJF-DROP-02` 现有点位与高德三期 POI 有约 0.5km 级差异，现场确认前不得视为准确服务位。
- [ ] 项目运营基地 `[121.080681,31.960337]` 先标为候选基地，补齐门牌、入口、装卸口和充电桩台账后再确认为运营基点。

### 2.2 范围基线

- [ ] `L1_CORE`：当前 V39 五个核心分区，作为默认无人车履约候选区。
- [ ] `L1_CONNECTOR`：主市场、A/B 城、三期、步行街，作为待核验连接区，默认不自动派单。
- [ ] `L0_COVERAGE`：叠石桥/川姜双中心 20km 分析范围，只用于统计和报表。
- [ ] `L2_EXTERNAL`：物流港、纺都大道 188 号、公共充电站等外部节点，单独建模。
- [ ] 旧 `REAL_RANGE` 不作为自动配送边界。
- [ ] 旧 `PILOT_RANGE` 只作为候选外包络，生产边界读取道路/服务位分区 Polygon。
- [ ] `parkWidthMeters=1570`、`parkHeightMeters=470` 仅代表 L1_CORE，不代表整个家纺城。

## 3. P0：先解决路线安全问题

### 3.1 禁止错误回退

- [ ] 地理路线失败时，地理配送车辆进入 `ROUTE_UNREACHABLE` / `ROUTE_BLOCKED` / `MANUAL_INTERVENTION`。
- [ ] `geoFollower` 为空时，地理配送车辆不得继续执行示意 `state.route`。
- [ ] 禁止真实车辆使用起点到终点直线作为可执行路线。
- [ ] 禁止 `STRAIGHT_LINE`、空路线、零长度路线进入车辆执行队列。
- [ ] 多车路线预约失败时，未锁定路线只能作为候选预览，不能标记为可执行。

### 3.2 修正空路线和起终点语义

- [ ] 空折线校验结果必须是无效/不可达，而不是有效。
- [ ] 起点必须来自车辆当前位置、有效服务位或道路接入节点。
- [ ] 终点必须来自取货、送货、充电或停车服务位。
- [ ] 起点和终点都必须吸附到道路边或服务位接入边。
- [ ] 吸附距离超过阈值时拒绝规划，不继续使用原始坐标。
- [ ] 起终点接入段也要进行建筑和车辆包络碰撞检查。

### 3.3 统一路线执行器

- [ ] 地理车辆只执行 GCJ-02 真实道路折线。
- [ ] 示意车辆和地理车辆使用明确的 `routeMode`，不得由编码前缀隐式决定。
- [ ] 路线执行绑定 `routeId`、`mapVersion`、`source`、`segmentPath`。
- [ ] 车辆沿折线弧长推进，记录当前路段、已行驶距离、方向角和时间戳。
- [ ] 车辆偏离路线超过阈值后暂停并重规划。

## 4. P0：补齐地图和道路数据

### 4.1 道路路段

- [ ] 每条路段补齐真实 `geometry`，不能只依赖两个节点直连。
- [ ] 补齐 `direction`、`widthMeters`、`roadClass`、`speedLimitKmh`。
- [ ] 补齐 `accessState`：可行驶、仅服务、步行、封闭、装卸、充电接入等。
- [ ] 补齐临时封路、门禁时间、限高、限宽、消防通道和禁停规则。
- [ ] 补齐转向限制、最小转弯半径和车辆类型限制。
- [ ] 道路、建筑和站点必须绑定同一 `mapVersion`。

### 4.2 建筑和障碍物

- [ ] 建立真实建筑物 Polygon，而不是只使用固定近似矩形。
- [ ] 建立围墙、河道、绿化隔离、施工区、停车障碍物和门禁点。
- [ ] 障碍物增加来源、版本、生效时间和硬禁行标识。
- [ ] 按车辆宽度、长度、朝向和安全缓冲膨胀障碍物。
- [ ] 检查路线中心线和车辆包络两层碰撞。
- [ ] 检查路线是否从建筑内部开始或结束。

## 5. P0：补齐站点和服务位

站点不是车辆终点，服务位才是车辆实际停靠目标。

- [ ] 每个取货/送货站点配置至少一个 `servicePosition`。
- [ ] 每个服务位配置 `coordLng`、`coordLat`、`accessNodeCode` 或 `accessEdgeId`。
- [ ] 配置 `serviceDirection`、`stopHeading`、进入方向和离开方向。
- [ ] 配置车型限制、车辆宽度/长度限制和服务时长。
- [ ] 配置门禁、时间窗、容量、排队容量和预约状态。
- [ ] 配置站点可信度：A 公共 POI、B 区域候选、C 内部合成点。
- [ ] 建筑中心、步行街中心、综合 POI 中心不得直接作为车辆目标。
- [ ] 公共充电 POI 与内部充电桩/停车位分成不同对象。
- [ ] 服务位未配置时，前端显示“无服务位”，后端返回不可达，不自动派车。

## 6. P1：车辆路由参数

每次路线规划必须读取或传入：

- [ ] `vehicleId`、`vehicleType`、`linkMode`。
- [ ] `widthCm`、`lengthCm`、`heightCm`、`turningRadiusM`。
- [ ] `safetyBufferMeters`、`allowedRoadClasses`、`allowedVehicleTypes`。
- [ ] `maxSpeedKmh`、`currentSpeedKmh`、`currentHeading`。
- [ ] `batteryLevel`、安全电量、充电能力和目标充电点。
- [ ] `currentMapVersion`、当前道路边或地图匹配结果。
- [ ] `manualOverride`、`emergencyMode`、人工接管状态。

已定义的 `VehicleRoutingProfile` 必须确认真正参与每次建图、寻路和碰撞校验，不能只停留在类型定义中。

## 7. P1：路线请求和结果契约

### 7.1 请求必须包含

- [ ] `parkId`、`mapVersion`、`coordSystem`。
- [ ] `originType`、`originLng`、`originLat`、起点服务位/接入节点。
- [ ] `destinationType`、`destinationLng`、`destinationLat`、终点服务位/接入节点。
- [ ] `routeMode=REAL_ROAD`。
- [ ] `allowStraightLine=false`。
- [ ] `avoidBuilding=true`、`avoidRiver=true`、`avoidPedestrianZone=true`。
- [ ] `snapDistanceMeters` 和起终点吸附结果。
- [ ] 路线来源策略：高德、本地路网或明确的受控回退顺序。

### 7.2 结果必须返回

- [ ] `routeId`、`mapVersion`、`source`、`routeMode`。
- [ ] `nodePath`、`segmentPath`、GCJ-02 `polyline`。
- [ ] 距离、行驶时间、等待时间、转弯数量和风险点。
- [ ] 起点/终点吸附距离、最大离路距离。
- [ ] `crossesBuilding`、`crossesRestrictedZone`、`vehicleFit`。
- [ ] `collisionChecked`、`reservationStatus`、`invalid`。
- [ ] `unreachableReason` 和可执行性说明。

## 8. P1：前端工作台调整

- [ ] 删除主界面底部“上线后的页面分工”板块。
- [ ] 主界面只保留任务队列、态势地图、资源风险、下一步动作。
- [ ] 地图图层改名为业务语义：L0 分析、L1 核心、L1 待核验连接、L2 外部。
- [ ] 地图默认隐藏低价值标签，点击后显示详情。
- [ ] 地图同时区分规划路线、实际轨迹、预测轨迹和回充路线。
- [ ] 右侧任务卡显示“可派原因/不可派原因/下一步动作”。
- [ ] 地图状态栏显示坐标系、地图版本、路线来源和更新时间。

## 9. P1：车辆和订单目标点视觉

- [ ] 车辆使用俯视简化车体、车头方向箭头和状态光环。
- [ ] 车辆状态至少区分空闲、配送、充电、装卸、等待、偏航、离线、人工接管。
- [ ] 车辆点击详情显示路线 ID、当前道路、路线来源、地图版本和偏航距离。
- [ ] 订单目标点使用“目标环 + 服务位芯点 + 道路接入虚线”。
- [ ] 取货、送货、充电、待命、快递接驳使用不同颜色和符号。
- [ ] 目标点显示可达、排队、占用、封闭、待核验状态。
- [ ] 目标点在建筑中心或无服务位时不绘制可执行路线。

## 10. P1：状态机、预约和能源

- [ ] 暂停派单时，自动派车、批量派车、紧急插队都受到统一控制。
- [ ] 任务状态迁移必须校验前置状态、车辆状态和路线状态。
- [ ] 自动派车、手动派车和紧急插队必须幂等，避免重复占用车辆。
- [ ] 服务位、充电桩和道路时空资源有统一锁定/释放机制。
- [ ] 取消、断线、路线失败、充电超时和人工接管都能释放资源。
- [ ] 充电决策纳入剩余电量、可达充电点、排队时间和下一任务可行性。
- [ ] 低电量车辆不能使用不可达或未核验的公共充电 POI。

## 11. P2：数据和运维质量

- [ ] 全仓库中文统一 UTF-8，修复前端和后端提示乱码。
- [ ] 所有路线、站点、地图和轨迹带时间戳与版本号。
- [ ] 建立路线健康指标：空路网、断连、未吸附、离路、穿建筑、直线兜底、预约冲突。
- [ ] 建立 10 条真实任务回放集，覆盖取货、送货、充电、封路和偏航。
- [ ] 每次路线规划保存审计信息，支持规划路线与实际轨迹对比。
- [ ] 地图数据发布前做节点、路段、站点、服务位、建筑和范围一致性检查。

## 13. 最终验收标准

- [ ] 路线的每一段都属于当前可行驶道路或已授权服务连接段。
- [ ] 车辆包络不进入建筑、河道、围墙和禁行区域。
- [ ] 订单目标必须落到服务位，不得落到建筑中心。
- [ ] 真实路线失败时车辆停止，不走示意路线。
- [ ] 路线版本、地图版本和车辆执行版本一致。
- [ ] 封路、门禁、车型不适配、服务位占用都会返回明确原因。
- [ ] 前端能区分规划路线、实际轨迹和不可执行路线。
- [ ] L0 分析范围不会让站点自动获得派单资格。
