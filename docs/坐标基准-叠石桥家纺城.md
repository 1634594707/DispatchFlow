# 坐标基准 · 叠石桥家纺城（ZJF 站点）

> 目的：用公开地理数据把 DispatchFlow 的 ZJF 站点坐标「定」在真实世界上。
> 结论先行：**项目现有 ZJF 坐标大体落在叠石桥家纺城 L1 核心试点范围内，坐标系选用 GCJ-02 + 高德(AMAP) 是正确的；但它们多数是项目内部调度锚点/候选服务位，不应全部表述为已由公开 POI 精确核验。** 主市场语义中心与 L1 局部投影基点需要分开记录，`ZJF-DROP-02` 仍需现场确认。

## 1. 坐标系框架（必须先统一认知）

| 系统 | 用途 | 本项目是否使用 |
| --- | --- | --- |
| **WGS-84** | GPS / OpenStreetMap / Leaflet 默认 | 仅作中间换算，不直接落库 |
| **GCJ-02**（火星） | 高德 / 腾讯 / 天地图底图 | ✅ **项目存储与底图统一用此系**（见 `V27` `map_provider='AMAP'`、`zjfStationAnchors.ts` 头注释） |
| **BD-09** | 百度底图 | 否（标准换算误差 ~80m，本项目用不到） |

**关键结论**：项目已统一 GCJ-02 存储 + 高德底图，**显示零漂移、无需运行时转换**。这为路线审查 4.6（坐标/距离混用）消去了最大的系统性隐患——只要后续不引入 WGS-84 实测点又直接做欧氏/Haversine 比较即可（见 §5）。

### 9.1 分成四个范围，不再使用一个半径表达全部业务

| 范围 | 建议定义 | 是否允许自动派单 | 参数 |
| --- | --- | --- | --- |
| **L1_CORE** | 当前 V39 的南排取货、北排仓库、代发仓、东排、快递接驳五个分区 | 是，前提是服务位和道路接入已核验 | 当前外包络约 1.57 km × 0.47 km |
| **L1_MARKET_CONNECTOR** | 家纺城主市场、A/B 城、三期、步行街等北侧市场连接区 | 暂不全量自动派单；仅允许已配置服务位的任务 | 建议单独建区，不扩大核心区默认派单 |
| **L0_INDUSTRY_COVERAGE** | 叠石桥与川姜/志浩双中心约 20 km 分析范围 | 否，仅用于统计、报表和范围提示 | `radiusMeters=20_000` 可保留，但名称必须标注“不参与派单” |
| **L2_EXTERNAL_LOGISTICS** | 纺都大道 188 号、深国际物流港、东南公共充电点等外部节点 | 否，除非单独配置物流接驳任务和道路权限 | 单独园区/外部路线，不混入 L1 图层 |

### 9.2 当前 L1 核心区参数是否合理

- `parkWidthMeters = 1570`：**合理**，覆盖当前 V39 五个核心分区的东西向外包络。
- `parkHeightMeters = 470`：**合理但只对 L1_CORE 有效**；不能宣称覆盖主市场北侧全部 POI。
- `anchor = [121.080354, 31.961977]`：**作为 L1 局部投影原点合理**，不作为主市场语义中心。
- `pilotPolygon` 旧矩形：**不应再作为生产配送边界**；它容易把建筑、步行街和非服务道路整体纳入“可配送区”。生产边界应使用五个道路/服务位分区多边形。
- `ZJF_L0_COVERAGE.radiusMeters = 20_000`：**作为分析范围合理**，作为自动派单范围不合理。

## 10. 坐标分级和读取参数建议

### 10.1 坐标可信度分级

- **A 级：公开 POI/地址已核验**：可用于地图展示和范围判断，不可直接代替服务位。
- **B 级：道路/区域候选点**：可用于路线规划预览，必须现场确认后才可自动派单。
- **C 级：项目合成的调度锚点/充电位/待命位**：只能作为内部模型数据，必须绑定接入道路节点和现场台账。

当前 13 个 `ZJF` 点大部分属于 **B/C 级**，不是 A 级公共 POI；项目文档此前“全部坐标已真实核验”的表述应改为“已完成区域级核验，服务位仍待现场核验”。

### 10.2 前端读取参数

- `mapCenter`：L1 默认读取 `[121.080354, 31.961977]`；L0 主市场概览读取 `[121.080800, 31.964305]`。
- `mapLevel`：L0/L1/L2 应代表业务范围，不再只代表技术图层。
- `parkWidthMeters/parkHeightMeters`：L1_CORE 使用 `1570/470`；L1_MARKET_CONNECTOR 单独计算尺寸，不直接覆盖原参数。
- `radiusMeters`：只给 L0 分析覆盖使用；L1 读取多边形，不读取半径。
- `stations`：读取业务站点和服务位两层对象；充电桩、停车位不能仅依赖站点编码前缀过滤。
- `route`：必须读取路线来源、地图版本、服务位 ID、接入节点和碰撞校验结果。
- `timestamp`：地图、站点、路网和车辆轨迹都要带更新时间，避免多版本叠加显示。

### 10.3 最终落库前的硬校验

- [ ] 坐标必须声明 `coordSystem=GCJ-02`。
- [ ] 站点必须有服务位或明确标记为“仅展示”。
- [ ] 服务位必须绑定 `accessNodeCode` 和 `serviceDirection`。
- [ ] 业务站点不能落在建筑物 Polygon 内。
- [ ] 路线首尾必须吸附到道路/服务位，不能把原始 POI 中心当作终点。
- [ ] `L0_INDUSTRY_COVERAGE` 内的站点默认不可自动派单。
- [ ] 公共充电 POI 与内部充电桩必须使用不同对象类型。
- [ ] `ZJF-DROP-02` 在现场确认前不得提升为 A 级坐标。

## 13. 本文结论

当前最稳妥的方案不是简单把地图中心移动到主市场 POI，也不是把所有高德 POI 都变成配送站点，而是：

1. 保留 `[121.080354,31.961977]` 作为 L1 核心区局部坐标基点。
2. 新增 `[121.080800,31.964305]` 作为 L0 主市场语义中心。
3. 继续使用 V39 五个核心分区做默认配送范围。
4. 把主市场、三期、步行街和公共充电站作为待核验的连接区/外部参考，不直接放开自动派单。
5. 将 `ZJF-DROP-02` 和全部内部充电位从“已确认”降为“区域级确认、服务位待现场确认”。
6. 将 20 km 半径保留为分析覆盖，不让它参与站点下单和自动派车。
